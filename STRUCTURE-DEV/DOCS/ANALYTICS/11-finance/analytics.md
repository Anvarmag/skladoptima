## 1. Цель аналитики раздела

Измерять полноту и достоверность управленческой юнит-экономики по товарам и периодам: выручка, расходы, прибыль, маржа, ROI. Аналитика нужна для решений по ассортименту, ценам, рекламе и приоритизации SKU.

---

## 2. Ключевые метрики (KPI)

| Метрика | Описание | Цель | Как считается |
|---------|---------|------|--------------|
| Unit Economics Coverage | Товары с полным расчетом | >= 80% | `products_with_complete_financial_data / products_in_finance_scope` |
| Negative Margin Share | Доля убыточных товаров | <= 20% | `products_margin_below_0 / products_with_finance` |
| Median Margin % | Медианная маржа по каталогу | Рост QoQ | `median(margin_pct_by_product)` |
| ROI Healthy Share | Доля товаров с ROI выше целевого порога | >= 60% | `products_roi_above_target / products_with_roi` |
| Data Completeness Warnings | Товары с warning неполного расчета | <= 25% | `products_with_warning / products_in_finance_scope` |
| Time to Cost Setup | Время до заполнения себестоимости нового SKU | <= 7 дней p50 | `median(cost_set_at - product_created_at)` |

---

## 3. Воронки и конверсии

```
SKU в каталоге -> Себестоимость задана -> Расчет метрик доступен -> SKU попал в управленческое решение
100%          -> 72%                 -> 68%                     -> 45%
```

Воронка предупреждений:

```
Warning incomplete data -> Пользователь открыл SKU -> Данные дозаполнены -> Warning снят
100%                    -> 55%                    -> 38%              -> 30%
```

## 6. Текущее состояние (baseline)

- Модуль работает, но baseline по полноте данных и качеству расчетов должен быть формализован в едином отчете.
- Главный baseline-артефакт: доля SKU без себестоимости и без ключевых расходных компонентов.
- На старте зафиксировать baseline по каждому marketplace отдельно, т.к. полнота расходов различается.

## 9. Риски и аномалии

| Аномалия | Порог | Действие |
|---------|-------|---------|
| Массовый рост warning incomplete | `> 35% SKU` | Проверить источники данных и модель обязательных полей |
| Резкое падение маржи по категории | `-10 п.п. WoW` | Разобрать комиссии, возвраты и рекламные расходы |
| Скачки ROI без изменений продаж | Любой нетипичный пик | Проверить корректность входных затрат |
| Доля убыточных SKU растет 2+ недели | `> 25%` | Эскалация в ассортиментные решения |

## 11. Источники данных и правила расчета

- Источники: `orders`, `marketplace financial reports`, `product cost profile`, `tenant tax settings`, ad cost feeds.
- `Unit Economics Coverage` считается только по SKU, попавшим в анализируемый период, а не по всему каталогу.
- Warnings должны различать тип недостающих данных: `cost`, `fees`, `logistics`, `tax`, `returns`, `ads`.
- Любой snapshot должен быть воспроизводим для одного и того же периода и набора источников.

---

## 12. Data Quality и QA-проверки

- Значения `profit`, `margin`, `roi` не должны вычисляться без traceable source inputs.
- QA должна проверить расчеты на SKU с полным набором данных и на SKU с отсутствующими cost/fees/logistics.
- Сумма детальных расходов не должна превышать общий расход отчета без объяснимой причины.
- Изменение cost profile должно отражаться в новых расчетах и быть видимым в audit trail.

## 13. Владельцы метрик и ритм ревью

- Product owner: прибыльность SKU и decision support.
- Finance/Product analyst: правила расчета и интерпретация warnings.
- Backend lead: reproducibility формул и snapshots.
- QA/Data: тесты корректности формул и мониторинг data completeness.

---

## 14. Зависимости, допущения и границы

- Финансовая аналитика не заменяет бухгалтерский учет и строится как управленческий слой по согласованным формулам.
- Для каждого marketplace должен быть зафиксирован стандарт состава расходов: комиссия, логистика, реклама, штрафы, возвраты, налоги.
- Если часть расходов приходит с лагом, в отчетах должен существовать статус свежести данных и флаг предварительного расчета.
- Любое сравнение периодов допустимо только при одинаковом наборе подключенных источников и одинаковой версии формул.
