## 1. Цель аналитики раздела

Обеспечить управленческий обзор продаж и ассортимента: динамика выручки, ABC по товарам, вклад маркетплейсов и рекомендации по ассортименту. Раздел должен помогать быстро принимать решения, а не перегружать пользователя сложной BI-логикой.

---

## 2. Ключевые метрики (KPI)

| Метрика | Описание | Цель | Как считается |
|---------|---------|------|--------------|
| Dashboard Adoption | Доля активных tenant, использующих модуль | >= 60% MAU | `tenants_opened_analytics / active_tenants` |
| Revenue Trend Stability | Наличие устойчивого week-over-week роста | Рост в целевых сегментах | `revenue_wow_change_pct` |
| ABC Concentration | Доля выручки от группы A | 60-80% | `revenue_group_A / total_revenue` |
| Top SKU Contribution | Вклад топ-10 SKU в выручку | Контроль концентрации | `revenue_top10 / total_revenue` |
| Recommendation Action Rate | Доля рекомендаций, по которым совершено действие | >= 25% | `recommendations_acted / recommendations_shown` |
| Drill-down Usage Rate | Пользователи, открывающие аналитику SKU из агрегата | >= 35% | `product_drilldown_opened / dashboard_sessions` |

---

## 3. Воронки и конверсии

```
Открытие dashboard -> Выбор периода -> Просмотр ABC -> Drill-down товара -> Принято действие
100%               -> 82%           -> 64%         -> 40%             -> 24%
```

Воронка рекомендаций:

```
Recommendation shown -> Recommendation opened -> Action executed
100%                 -> 55%                  -> 25%
```

## 9. Риски и аномалии

| Аномалия | Порог | Действие |
|---------|-------|---------|
| Падение использования модуля | `< 40% активных tenant` | Пересмотреть структуру дашборда и навигацию |
| Слишком высокая концентрация группы A | `> 85% выручки` | Риск зависимости; рекомендовать диверсификацию |
| Низкая реакция на рекомендации | `< 15%` | Пересмотреть rule-based правила и приоритеты |
| Расхождения между продажами и аналитикой | Любой системный кейс | Проверить пайплайн источников и агрегации |

## 11. Источники данных и правила расчета

- Источники: `orders`, `order_items`, materialized daily aggregates, product metadata, optional finance enrichments.
- Revenue dashboard и ABC должны считать одинаковую базовую выручку за один и тот же период.
- ABC-анализ строится по net или gross revenue только после явной фиксации стандарта; смешивать нельзя.
- Recommendation engine должен хранить версию rule-set, иначе нельзя интерпретировать исторические изменения.

---

## 12. Data Quality и QA-проверки

- Drill-down по SKU должен биться с агрегатом dashboard по тому же периоду.
- QA должна проверить пустой tenant, короткий период, длинный период, одинаковую выручку у нескольких SKU, export.
- В daily aggregate не должно быть пропусков дат для активного периода отображения.
- Возвраты и net revenue должны считаться единообразно во всех виджетах раздела.

## 13. Владельцы метрик и ритм ревью

- Product owner: adoption дашборда и actionability рекомендаций.
- Data/Analytics owner: корректность агрегатов и ABC-логики.
- Backend lead: materialized views/snapshots и performance.
- QA: consistency dashboard vs drill-down vs export.

---

## 14. Зависимости, допущения и границы

- Раздел аналитики использует согласованный источник правды по заказам и выручке; параллельные “быстрые” расчеты в UI недопустимы.
- ABC-классификация должна пересчитываться по стабильному правилу накопленной доли, а не по произвольным порогам на уровне клиента.
- Рекомендации внутри модуля не считаются ML и должны быть объяснимыми через rule-based причину и входные данные.
- Для крупных tenant требуется ограничение на тяжелые ad-hoc запросы: пользователь должен видеть snapshot/read-model, а не прямой запрос к OLTP.
