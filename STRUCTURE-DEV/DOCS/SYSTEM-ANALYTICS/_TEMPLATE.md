## 1. Назначение

_Кратко: зачем модуль нужен бизнесу и продукту, какую доменную ответственность несет._

## 2. Функциональный контур и границы

_Что входит в модуль, что не входит, где границы ответственности относительно соседних модулей._

## 3. Акторы и зоны ответственности

_Кто пользуется модулем, какие роли есть, кто инициирует процессы, кто внешний системный actor._

## 4. Базовые сценарии использования

_Основные пользовательские и системные сценарии: как функционал реально работает пошагово._

## 5. Зависимости и интеграции

_Какие внешние модули/сервисы нужны модулю для работы._

## 6. API-контракт (внедрить)

_Какие endpoint нужно реализовать. Для каждого — метод, путь, авторизация, назначение._

## 7. Примеры вызова API

_Примеры `curl`/request/response для must-have сценариев._

## 8. Модель данных (PostgreSQL)

_Сущности, поля, индексы, уникальности, FK, soft delete, статусные поля._

## 9. Сценарии и алгоритмы (step-by-step)

_Алгоритмы обработки данных, orchestration, порядок шагов, важные инварианты._

## 10. Валидации и ошибки

_Правила валидации входных данных и согласованная error taxonomy._

## 11. Чеклист реализации

_Конкретный инженерный scope для BE/FE/DB/Infra/QA._

## 12. Критерии готовности (DoD)

_Что должно быть выполнено, чтобы модуль считался реально готовым к релизу._

## 13. Специальные системные правила модуля

_RBAC, state machine, специальные политики, lifecycle, routing rules — все module-specific правила, которые нельзя потерять._

## 14. Async процессы / Events / Webhooks

_Какие фоновые процессы, события, очереди, outbox, scheduler, webhooks используются._

## 15. Фазы внедрения

_Реалистичная последовательность поставки, чтобы модуль можно было строить инкрементально._

## 16. Тестовая матрица

_Что обязательно покрыть unit/integration/e2e/regression тестами._

## 17. Нефункциональные требования и SLA

_Производительность, доступность, безопасность, идемпотентность, изоляция, TTL, concurrency._

## 18. Observability, логи и алерты

_Что логировать, какие метрики собирать, на что ставить алерты, какие operational dashboards нужны._

## 19. Риски реализации и архитектурные замечания

_Что может пойти не так, какие решения критичны до старта разработки, где возможны ловушки в архитектуре._
