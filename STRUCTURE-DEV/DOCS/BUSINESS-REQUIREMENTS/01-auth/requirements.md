1. Описание раздела

Раздел определяет правила:

регистрации пользователя
подтверждения email
входа в систему
выхода из системы
восстановления пароля
смены пароля
управления пользовательской сессией
доступа пользователя в приложение после входа
первичного входа пользователя без компании
переключения между компаниями, если пользователь состоит в нескольких компаниях
2. Целевая аудитория
Роль	Доступные действия
Гость	Регистрация, подтверждение email, вход, восстановление пароля
Авторизованный пользователь	Выход, смена пароля, доступ в приложение, создание компании, принятие приглашения
Пользователь без компании	Вход в систему, прохождение онбординга, создание компании или присоединение по приглашению
3. Бизнес-цели
Обеспечить безопасный и понятный вход пользователей в систему.
Исключить регистрацию с недостоверными email.
Обеспечить уникальность пользователя по email и phone.
Создать основу для многокомпанейной модели, где один пользователь может работать в нескольких компаниях.
Позволить пользователю начать работу даже без созданной компании.
Подготовить фундамент для будущего расширения через SMS login, Google login и Yandex login.
Обеспечить безопасное управление сессиями и восстановлением доступа.
4. Основные сущности
4.1 User

Аккаунт пользователя в системе.

4.2 Company / Tenant

Компания или рабочее пространство, в рамках которого пользователь работает с данными.

4.3 Membership

Связь пользователя с конкретной компанией. Через membership определяется роль и доступ пользователя внутри компании.

4.4 Auth Session

Активная пользовательская сессия в браузере.

4.5 Invitation

Приглашение пользователя в компанию.

4.6 Auth Identity

Способ входа пользователя в систему:

local auth - email + password
future - phone + SMS
future - Google
future - Yandex
5. Базовые бизнес-правила
Один email может принадлежать только одному user.
Один phone может принадлежать только одному user.
Пользователь регистрируется по email, phone и password.
Пользователь получает полный доступ в систему только после подтверждения email.
Пользователь может существовать в системе без компании.
Один пользователь может состоять в нескольких компаниях.
Доступ пользователя к компании определяется через membership.
Роль пользователя определяется в рамках membership, а не глобально на уровне user.
Компания создается пользователем вручную после входа в систему.
Пользователь может присоединиться к компании по приглашению.
Основной способ приглашения сотрудника - по email.
При смене пароля все остальные активные сессии пользователя должны завершаться.

8. Логика регистрации
Пользователь открывает экран регистрации.
Пользователь вводит email, phone и password.
Система проверяет корректность заполнения полей.
Система проверяет уникальность email и phone.
Если данные валидны, система создает user в статусе ожидания подтверждения email.
Система отправляет письмо с одноразовой ссылкой подтверждения.
Пользователь переходит по ссылке из письма.
Система подтверждает email.
После подтверждения email пользователь переходит на экран входа.
Email может быть автоматически подставлен в форму входа.
9. Логика входа
Пользователь открывает экран входа.
Вводит email и password.
Система проверяет корректность учетных данных.
Система проверяет, подтвержден ли email.
Если email не подтвержден, пользователь не получает полный доступ в систему и должен завершить подтверждение.
Если все проверки успешны, создается пользовательская сессия.
После входа:
если у пользователя нет компаний - показывается онбординг
если у пользователя одна компания - открывается она
если у пользователя несколько компаний - открывается last used company
если last used company недоступна - показывается экран выбора компании
10. Логика email verification
Подтверждение email выполняется по одноразовой ссылке.
Ссылка имеет ограниченный срок действия.
Если срок действия ссылки истек, пользователь видит сообщение об истечении и может запросить новую ссылку.
Если email уже подтвержден, пользователь видит нейтральное сообщение "Email уже подтвержден".
Пользователь может повторно запросить verification email с ограничением по частоте отправки.
Дефолтные ограничения
cooldown на повторную отправку - 60 секунд
не более 3 повторных отправок в течение 1 часа
срок жизни verification link - 24 часа
11. Логика восстановления пароля
Пользователь нажимает "Забыли пароль?".
Вводит email.
Система отправляет письмо со ссылкой для сброса пароля.
Пользователь переходит по ссылке.
Вводит новый пароль.
Система сохраняет новый пароль.
Все остальные активные сессии пользователя завершаются.

Комментарий: точный срок жизни reset password link будет зафиксирован позже.

12. Логика приглашения в компанию
OWNER приглашает сотрудника по email.
Сотруднику отправляется письмо со ссылкой-приглашением в приложение.
Ссылка приглашения содержит invitation token.
Если пользователь с таким email уже существует, после принятия приглашения ему добавляется membership в соответствующую компанию.
Если пользователя с таким email еще нет, он проходит регистрацию и подтверждение email, после чего получает membership в приглашенную компанию.
Если срок действия приглашения истек, пользователь видит сообщение об истечении срока и должен запросить новое приглашение у владельца компании.
Дефолтное ограничение
срок жизни invitation link - 7 дней
13. Логика пользователя без компании
Пользователь может существовать в системе без компании.
После входа такой пользователь попадает в онбординг.
В онбординге доступны действия:
создать компанию
присоединиться к компании по приглашению
Полноценный trial-функционал доступен только после создания компании или присоединения к ней.
14. Политика сессий
Пользовательская сессия создается после успешного входа.
Для web-сессии используется access token + refresh session.
Logout завершает текущую сессию.
При смене пароля все остальные активные сессии инвалидируются.
Для MVP отдельный экран списка устройств не является обязательным.
Рекомендуемые дефолты для аналитики и реализации
access token TTL - 15 минут
refresh session TTL - 7 дней
15. Ограничения и anti-abuse правила
Повторная отправка verification email ограничивается.
Между повторными отправками действует cooldown.
После превышения лимита повторная отправка временно блокируется.
Неподтвержденный аккаунт не должен жить в системе бесконечно.
Без подтвержденного email пользователь не должен попадать внутрь основного приложения.
Рекомендуемые дефолты
cooldown resend - 60 секунд
не более 3 повторных отправок в течение 1 часа
хранение неподтвержденного аккаунта - 30 дней
16. Нефункциональные требования
Требование	Значение
Безопасность cookie	httpOnly, secure
Хранение паролей	только в хэшированном виде
Защита от brute force	rate limiting на login / reset / resend
Защита приватных маршрутов	обязательна
Серверная инвалидация сессий	обязательна
Поддержка мультитенантности	обязательна
Поддержка audit log auth-событий	обязательна
17. Ошибки и пользовательские состояния

Система должна корректно обрабатывать сценарии:

неверный email или password
email не подтвержден
verification link истек
verification link уже использован
email уже подтвержден
приглашение истекло
приглашение уже использовано

сессия истекла

остальные сессии завершены
