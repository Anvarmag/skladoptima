1. Описание раздела

Раздел определяет:

бизнес-аудит значимых действий в системе
историю изменений по ключевым сущностям
базовые security events
просмотр, фильтрацию и детализацию аудита
разграничение доступа к аудиту по ролям
правила хранения и отображения аудита

На MVP аудит предназначен для прозрачности действий пользователей и системы, расследования ошибок, контроля изменений и повышения доверия к данным.

2. Целевая аудитория
Роль	Действия доступны
Primary Owner	Просмотр полного аудита tenant
Admin	Просмотр полного аудита tenant
Manager	Просмотр operational audit по каталогу, складам и остаткам
Staff	Нет доступа к аудиту
SUPPORT_ADMIN	Просмотр полного аудита по tenant для служебной поддержки
3. Бизнес-цели
Дать owner и admin прозрачную историю значимых действий в системе.
Обеспечить возможность расследовать расхождения по остаткам, ролям, подключениям и настройкам.
Поддержать поддержку и локализацию проблем через служебный доступ SUPPORT_ADMIN.
Зафиксировать неизменяемую историю критичных изменений.
Разделить бизнес-аудит и технические логи системы.
4. Базовые бизнес-правила
Бизнес-аудит и технические sync-логи являются разными слоями и должны храниться отдельно.
В аудит попадают только значимые бизнесовые и базовые security-события.
Записи аудита на MVP являются immutable:
их нельзя редактировать вручную
их нельзя удалять вручную через UI
Manager не должен видеть billing-аудит, security-аудит других пользователей и административные изменения вне своей операционной зоны.
SUPPORT_ADMIN может видеть полный аудит по tenant для диагностики и поддержки.
На MVP срок доступного просмотра аудита может зависеть от тарифа как правило отображения/доступа, а не обязательно как физическое удаление записей.
Экспорт аудита не является обязательной функцией MVP.
5. Разделение уровней логирования
5.1 Business Audit

События, важные для пользователя и бизнеса:

кто что изменил
что было до
что стало после
когда это произошло
5.2 Security Events

Базовые события безопасности:

login success
failed login
password reset request
password changed
5.3 Technical Logs

Технические sync-логи и системные логи не входят в бизнес-аудит и ведутся отдельно.

FR-11: Система должна поддерживать разделение прав доступа к аудиту по ролям.
FR-12: Система должна сохранять immutable записи аудита.
FR-13: Система должна поддерживать фильтр по источнику события.
FR-14: Система должна разделять бизнес-аудит и технические sync-логи.
Should Have

14. Retention и доступ к истории

На MVP доступ к истории аудита может зависеть от тарифа как правило отображения/доступа.

Пример будущей политики
Basic - 30 дней
Pro - 90 дней
Business - 365 дней

На MVP это правило трактуется прежде всего как ограничение доступной глубины просмотра, а не как обязательная сложная политика физического удаления данных.

15. Security Events на MVP

На MVP включаются базовые security events:

login success
failed login
password reset request
password changed
Не входит в MVP
подозрительные кейсы уровня SuspiciousCase
сложная поведенческая security-аналитика
автоматические расследовательские сценарии
16. Ограничения и правила
Audit log не должен быть свален в один поток с техническими sync-логами.
Аудит должен быть ориентирован на понятность для бизнеса и поддержки.
Immutable-записи не редактируются и не удаляются вручную через UI.
Не все роли видят одинаковый объем аудита.
Аудит должен помогать расследовать спорные изменения, а не служить техническим трассировочным логом.
17. Что не входит в этот раздел

Чтобы не размывать границы, в этот раздел не входят:

технические sync-логи
worker/internal debug logs
advanced suspicious case engine
сложная корреляция security-инцидентов
полная forensic/infosec-система
расширенная экспортная и отчетная подсистема по аудиту
18. Комментарии на будущее
В будущем можно добавить экспорт аудита.
В будущем можно добавить более глубокие security events.
В будущем можно добавить suspicious cases и расследовательские сценарии.
