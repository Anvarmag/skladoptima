1. Описание раздела

Раздел определяет:

подключение маркетплейсов к tenant
управление учетными данными и идентификаторами подключений
проверку валидности подключений
статусы подключений и ошибки интеграции
отключение подключений без удаления исторических данных
работу с несколькими marketplace accounts внутри одного tenant

MarketplaceAccount представляет собой отдельную сущность подключения внешнего канала к tenant, а не просто поле с API-ключом.

2. Целевая аудитория
Роль	Действия доступны
Primary Owner	Полное управление подключениями
Admin	Добавление, редактирование, перепроверка и отключение подключений
Manager	Может видеть статус подключений без доступа к секретам
Staff	Не видит раздел подключений
3. Бизнес-цели
Обеспечить безопасное подключение маркетплейсов к tenant.
Обеспечить стабильную синхронизацию данных между SkladOptima и внешними каналами.
Поддержать несколько marketplace accounts в рамках одного tenant.
Исключить доступ неуполномоченных ролей к секретам подключения.
Обеспечить прозрачную диагностику статуса подключения и ошибок интеграции.
Сохранить исторические данные и связи при отключении канала.
4. Поддерживаемые маркетплейсы на MVP

На MVP поддерживаются:

Wildberries
Ozon
Яндекс.Маркет

В будущем список маркетплейсов и внешних каналов может расширяться.

5. Основные сущности
5.1 MarketplaceAccount

Сущность подключения внешнего канала к tenant.

5.2 Marketplace Credentials

Набор секретов и идентификаторов, необходимых для работы с API конкретного маркетплейса.

5.3 Marketplace Connection Status

Статус текущего состояния подключения.

5.4 Sync Metadata

Метаданные синхронизации подключения:

время последней проверки
время последней синхронизации
состояние активности
ошибка подключения или sync, если есть
6. Базовые бизнес-правила
Один tenant может иметь несколько marketplace accounts.
Один tenant может иметь несколько подключений одного и того же маркетплейса.
Подключениями на MVP управляют только Primary Owner и Admin.
Manager может видеть факт подключения и статус, но не видит секреты.
Staff не видит раздел подключений.
Секреты подключения не должны передаваться на фронтенд в plaintext.
После сохранения секреты могут отображаться только в маскированном виде.
Ошибка одного marketplace account не должна останавливать работу других подключений tenant.
При отключении подключения исторические данные и ранее загруженные данные сохраняются.
Отключение подключения означает остановку синхронизации, а не удаление данных.
Физическое удаление подключения на MVP не поддерживается.
На MVP лимиты подключений по тарифу не вводятся.
7. Пользовательские сценарии
US-01

Как Primary Owner, я хочу подключить Wildberries, чтобы включить синхронизацию данных.

US-02

Как Primary Owner, я хочу подключить Ozon, чтобы получать товары, остатки и другие данные.

US-03

Как Primary Owner, я хочу подключить Яндекс.Маркет, чтобы работать с данными этого канала в SkladOptima.

US-04

US-05

Как пользователь с правами управления подключениями, я хочу отключить marketplace account без потери истории, чтобы остановить синхронизацию.

US-06

Как Manager, я хочу видеть статус подключения маркетплейсов, чтобы понимать, работают ли интеграции.

10. Проверка и валидация подключения
При сохранении подключения система должна выполнять проверку валидности credentials.
Если credentials невалидны, подключение не должно считаться активным рабочим подключением.
Система должна показывать пользователю ошибку валидации.
Если credentials формально валидны, но прав недостаточно, система должна помечать подключение как проблемное и уведомлять пользователя.
Пользователь должен иметь возможность выполнить повторную проверку подключения вручную.
11. Статусы подключения

На MVP рекомендуется использовать следующие статусы:

connected
invalid_credentials
inactive
sync_error
needs_reconnect
Описание
connected - подключение активно и работает
invalid_credentials - credentials невалидны
inactive - подключение отключено пользователем
sync_error - произошла ошибка синхронизации
needs_reconnect - подключение требует обновления credentials или переподключения
12. Логика отключения подключения
Отключение подключения не удаляет исторические данные.
Отключение подключения останавливает синхронизацию по данному каналу.
Связи товаров и ранее загруженные данные сохраняются.
Другие подключенные маркетплейсы продолжают работать независимо.
После повторного подключения или обновления credentials система должна иметь возможность возобновить работу с актуальными данными.
13. Безопасность
Credentials должны храниться только в защищенном виде.
Полные значения credentials не должны передаваться на фронтенд после сохранения.
В интерфейсе допускается отображение только частично маскированного значения для идентификации пользователем.
Доступ к credentials имеют только роли с правами управления подключениями.
Действия с подключениями должны логироваться.
14. История и аудит

Система должна сохранять историю действий по подключению, как минимум:

подключение создано
credentials обновлены
подключение отключено
подключение перепроверено
validation failed
sync error detected

История подключений может использовать общую систему audit/history платформы.

статус подключения

время последней проверки
время последней синхронизации
Права отображения

Manager видит только факт наличия подключения и его статус
Staff не видит раздел
16. Ограничения и правила
Подключение маркетплейса не является просто полем API key.
Один tenant может иметь несколько marketplace accounts.
Секреты не показываются в открытом виде после сохранения.
Ошибка одного подключения не должна влиять на другие подключения tenant.
Отключение подключения не должно удалять данные и историю.
Физическое удаление подключения не входит в MVP.
Лимиты подключений на MVP не вводятся.
17. Что не входит в этот раздел

Чтобы не размывать границы, в этот раздел не входят:

детальная логика синхронизации товаров, остатков, заказов и отзывов
логика inventory sync как отдельный модуль
лимиты подключений по тарифу
сложная диагностика scopes / permissions
массовое подключение нескольких аккаунтов за один сценарий
расширенная диагностика API-методов
поддержка OAuth-like flows с refresh lifecycle
автоматическое исправление проблемных подключений
