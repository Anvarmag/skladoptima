1. Описание раздела

Раздел определяет:

приглашение пользователей в tenant
принятие приглашения
создание и управление membership
просмотр состава команды tenant
изменение роли участника команды
отзыв доступа участника к tenant
ограничения по количеству участников в зависимости от тарифа

Membership представляет собой связь пользователя с конкретным tenant и определяет его роль и статус доступа в рамках этой компании.

2. Целевая аудитория
Роль	Доступные действия
Primary Owner	Полное управление составом команды tenant
Admin	Просмотр активного состава команды
Manager	Просмотр состава команды
Staff	Нет доступа к управлению командой
Приглашенный пользователь	Принятие приглашения, вход или регистрация для получения доступа
3. Бизнес-цели
Обеспечить безопасное добавление сотрудников в tenant.
Обеспечить управляемую связь между user и tenant через membership.
Исключить несанкционированный доступ к tenant.
Поддержать базовую ролевую модель команды на MVP.
Обеспечить возможность отзыва доступа сотрудников без удаления их глобального user-аккаунта.
Поддержать лимиты по количеству участников в зависимости от тарифа tenant.
4. Основные сущности
4.1 Membership

Связь пользователя с tenant, содержащая:

пользователя

роль
статус доступа
дату приглашения
дату присоединения
дату отзыва доступа
инициатора приглашения

Приглашение пользователя в tenant по email, содержащее invitation token и срок действия.

5. Базовые бизнес-правила
Один user может состоять в нескольких tenant.
Один tenant может иметь нескольких пользователей.
Доступ пользователя к tenant определяется membership.
Роль пользователя определяется в рамках membership, а не глобально.
Приглашение в tenant выполняется только по email.
Если пользователь с таким email уже существует, ему не создается новый user-аккаунт.
Если пользователя с таким email еще нет, он проходит регистрацию и после подтверждения email получает membership в tenant.
При отзыве доступа удаляется или деактивируется membership, а не глобальный user-аккаунт.
В одном tenant на MVP существует один Primary Owner.
Primary Owner не может быть удален из tenant на MVP.
Передача роли Primary Owner другому пользователю не входит в MVP.
Tenant не должен оставаться без активного Primary Owner.
Primary Owner не может покинуть tenant на MVP.
Сотрудники с ролями Admin, Manager и Staff могут самостоятельно покинуть tenant.
6. Пользовательские сценарии
US-01

Как Primary Owner, я хочу пригласить сотрудника по email, чтобы он получил доступ к моей компании.

US-02

Как Primary Owner, я хочу видеть состав команды tenant, чтобы управлять доступом сотрудников.

US-03

Как Primary Owner, я хочу изменить роль участника tenant, чтобы скорректировать его уровень доступа.

US-04

Как Primary Owner, я хочу отозвать доступ участника, чтобы заблокировать его доступ к tenant.

US-05

Как приглашенный пользователь, я хочу перейти по ссылке из email, войти или зарегистрироваться и получить доступ к tenant.

US-06

Как сотрудник tenant, я хочу при необходимости самостоятельно покинуть компанию, чтобы убрать свой доступ к tenant.

7. Роли в рамках tenant

На MVP используются следующие роли:

PRIMARY_OWNER
ADMIN
MANAGER
STAFF
Бизнес-смысл ролей
PRIMARY_OWNER

Главный владелец tenant. Управляет командой, ролями, критичными настройками tenant и основными административными действиями.

ADMIN

Доверенный пользователь с почти полным доступом к рабочим функциям tenant, но без прав на управление составом команды, ролями и критичными owner-действиями.

MANAGER

Пользователь с ограниченным операционным доступом. Может работать в рамках назначенных задач и просматривать состав команды, но не управляет участниками.

STAFF

Пользователь с минимальным или ограниченным доступом, предназначенный для простых рабочих сценариев и просмотра/базовых операций.

Управление командой на MVP
PRIMARY_OWNER может приглашать ADMIN, MANAGER, STAFF
PRIMARY_OWNER может изменять роли участников tenant
PRIMARY_OWNER может отзывать доступ участников tenant
ADMIN не приглашает сотрудников на MVP
ADMIN не изменяет роли на MVP
ADMIN не отзывает доступ на MVP
MANAGER может просматривать состав команды
STAFF не управляет командой

Комментарий: детальная матрица прав по разделам продукта выносится в отдельный модуль RBAC / Permissions.

Must Have

FR-02: Система должна создавать invitation token для приглашения в tenant.
FR-03: Система должна поддерживать страницу принятия приглашения.
FR-04: Система должна определять, существует ли пользователь с email приглашения.
FR-05: Если пользователь уже существует, система должна позволять ему войти и получить membership в tenant.
FR-06: Если пользователь еще не существует, система должна позволять ему зарегистрироваться и после подтверждения email создать membership в tenant.
FR-07: Система должна поддерживать список участников команды tenant.
FR-08: Система должна отображать роль и статус участника tenant.
FR-09: Система должна позволять Primary Owner изменять роль участника tenant.
FR-10: Система должна позволять Primary Owner отзывать доступ участника tenant.
FR-11: Система должна поддерживать membership как отдельную сущность user-to-tenant связи.
FR-12: Система должна запрещать удаление Primary Owner из tenant на MVP.
FR-13: Система должна позволять сотруднику с ролью Admin, Manager или Staff самостоятельно покинуть tenant.
FR-14: После принятия приглашения система должна открывать именно тот tenant, в который был приглашен пользователь.
FR-15: Система должна учитывать лимит участников tenant в зависимости от тарифа.

FR-17: Система должна поддерживать отмену непринятого приглашения.
FR-18: Система должна отображать истекшее или уже использованное приглашение корректным пользовательским сообщением.
FR-19: Система должна отображать участникам команды дату приглашения и дату вступления там, где это необходимо по роли и интерфейсу.
FR-20: Система должна показывать сообщение о достижении лимита участников и предлагать расширение лимита за доплату.

FR-22: Система должна позволять Admin отзывать доступ пользователей с ограниченным набором ролей.

9. Логика приглашения
Primary Owner открывает раздел управления командой tenant.
Указывает email приглашенного пользователя.
Выбирает роль для нового участника.
Система проверяет, не превышен ли лимит участников tenant.
Система создает приглашение и invitation token.
Система отправляет email со ссылкой-приглашением.
Приглашенный пользователь переходит по ссылке.
Если у него уже есть аккаунт, он входит в систему.
Если аккаунта нет, он регистрируется и подтверждает email.
После успешного входа или регистрации система создает membership в tenant.
Пользователь получает доступ к tenant в рамках назначенной роли.
После принятия приглашения открывается именно тот tenant, куда пользователя пригласили.
Дефолтные ограничения
срок жизни invitation link - 7 дней
при повторной отправке приглашения используется новый token
в одном tenant не должно быть нескольких одновременно активных pending invites на один и тот же email
если email уже active member в tenant, повторный invite не создается
если пользователь ранее был removed из tenant, его можно пригласить повторно
Рекомендуемые ограничения на resend
cooldown между повторными отправками - 60 секунд
не более 3 повторных отправок в течение 1 часа на один email в одном tenant
10. Статусы membership

На MVP используются следующие статусы:

invited - приглашение отправлено, но еще не принято
active - пользователь состоит в tenant и имеет доступ
removed - доступ пользователя к tenant отозван или пользователь самостоятельно покинул tenant
11. Логика изменения роли
Изменение роли участника tenant выполняет только Primary Owner.
Изменение роли применяется только в рамках конкретного tenant.
Глобальный user-аккаунт при этом не изменяется.
Primary Owner может повышать и понижать роли участников tenant.
Пользователь не может изменять собственную роль.
Изменение роли Primary Owner на MVP не поддерживается.
12. Логика отзыва доступа
Отзыв доступа участника означает удаление или деактивацию membership в конкретном tenant.
После отзыва доступа пользователь теряет доступ к этому tenant.
При этом user-аккаунт пользователя сохраняется в системе.
Если у пользователя есть доступ к другим tenant, они сохраняются.
Отзыв доступа участника выполняет только Primary Owner.
Удаление Admin, Manager или Staff является отзывом membership.
Primary Owner не может быть отозван или удален из tenant на MVP.
13. Логика самостоятельного выхода из tenant
Пользователь с ролями Admin, Manager и Staff может самостоятельно покинуть tenant.
После самостоятельного выхода его membership переводится в статус removed.
После выхода пользователь теряет доступ к данному tenant.
Если у пользователя есть доступ к другим tenant, они сохраняются.
Primary Owner не может самостоятельно покинуть tenant на MVP.
14. Лимиты участников по тарифу
Количество участников tenant ограничивается тарифом.
На MVP базовый лимит составляет:
Primary Owner
Admin
Manager
То есть без дополнительной оплаты tenant может иметь до 3 участников.
Trial использует тот же лимит, что и базовый стартовый сценарий.
При достижении лимита новый участник не может быть добавлен.
При попытке добавить участника сверх лимита система должна показывать сообщение о достижении лимита и предлагать расширение за доплату.

Комментарий: детальная логика тарифов и стоимости расширения лимитов выносится в отдельный модуль Billing / Subscription.

16. Ограничения и правила
Нельзя пригласить пользователя в tenant без указания роли.
Нельзя создать дублирующий user-аккаунт по email приглашения.
Нельзя создать несколько активных pending invites на один и тот же email в одном tenant.
Нельзя удалить Primary Owner из tenant на MVP.
Нельзя передать роль Primary Owner на MVP.
Нельзя менять собственную роль.
Нельзя оставить tenant без активного Primary Owner.
Истекшее приглашение не дает доступ к tenant.
Team management не должен смешиваться с глобальным управлением user-аккаунтом.
Team / Membership не должен смешиваться с реферальной или партнерской программой.
17. Нефункциональные требования
Требование	Значение
Безопасность приглашений	одноразовый invitation token с ограниченным сроком действия
Изоляция tenant	membership действует только внутри конкретного tenant
Уникальность user	один email = один user
Аудит	действия по приглашению, изменению роли и отзыву доступа должны логироваться
Anti-spam	resend invite должен быть ограничен по частоте
18. Что не входит в этот раздел

Чтобы не размывать границы, в этот раздел не входят:

регистрация и логин пользователя
детальная матрица permissions по модулям
биллинг и лимиты тарифов как полноценный модуль
передача роли Primary Owner другому пользователю
кастомные роли
шаблоны ролей
приглашение по публичной ссылке
самостоятельное вступление в tenant по коду
approval flow для вступления в tenant
отдельные роли бухгалтер и аналитик
массовые приглашения через CSV / Excel
расширенный аудит изменений команды в интерфейсе
поиск и фильтрация по участникам команды
