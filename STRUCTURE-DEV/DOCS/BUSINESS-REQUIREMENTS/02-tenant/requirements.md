1. Описание раздела

Раздел определяет:

модель tenant как отдельного аккаунта компании в SaaS
правила создания tenant
правила изоляции данных между tenant
жизненный цикл доступа tenant к системе через AccessState
базовые настройки tenant
правила блокировки, закрытия и хранения данных tenant

Tenant представляет собой отдельное юридическое лицо в системе.
Каждый tenant логически изолирован от других tenant.

2. Целевая аудитория
Роль	Доступные действия
Пользователь без компании	Создание нового tenant
Primary Owner	Просмотр и изменение базовых настроек tenant, управление tenant в рамках своих прав
SUPPORT_ADMIN	Управление AccessState tenant через внутренние административные инструменты

Примечание: SUPPORT_ADMIN - внутренняя служебная роль платформы, не роль клиентского пользователя.

3. Бизнес-цели
Обеспечить изолированную работу нескольких компаний в одной SaaS-платформе.
Поддержать модель, в которой один пользователь может состоять в нескольких tenant.
Зафиксировать tenant как владельца подписки, доступа и бизнес-настроек компании.
Обеспечить управляемый жизненный цикл tenant от создания до закрытия.
Исключить смешивание данных, действий и доступа между tenant.
Подготовить основу для trial, ограничения доступа и будущей тарификации.
4. Основные сущности
4.1 Tenant

Юридическое лицо или отдельный бизнес-аккаунт клиента в системе.

4.2 Membership

Связь пользователя с tenant, определяющая доступ пользователя к конкретной компании.

4.3 AccessState

Бизнес-состояние доступа tenant к системе, связанное с trial, оплатой, ограничением доступа или закрытием.

4.4 Tenant Settings

Базовые настройки tenant, включая реквизиты и операционные параметры компании.

5. Базовые бизнес-правила
Один tenant соответствует одному юридическому лицу.
В рамках одного tenant могут существовать несколько магазинов, брендов, складов и подключений маркетплейсов.
Один user может состоять в нескольких tenant.
Один user может создать несколько tenant.
Один tenant может иметь нескольких пользователей.
Tenant не создается автоматически при регистрации пользователя.
Tenant создается только после входа пользователя в систему.
Tenant создается явным действием пользователя.
При создании tenant пользователь автоматически получает роль Primary Owner.
На MVP в одном tenant существует один Primary Owner.
Передача роли Primary Owner другому пользователю не входит в MVP.
Подписка и доступ принадлежат tenant, а не user.
Доступ к данным всегда определяется в контексте конкретного tenant.
Все данные tenant должны быть логически изолированы от других tenant.
При SUSPENDED данные tenant не удаляются.
При CLOSED доступ tenant прекращается, а данные переводятся в режим retention до окончательного удаления или архивирования по политике хранения.
Возможность объединения нескольких юрлиц внутри одного tenant не входит в MVP и рассматривается как future scope.
6. Пользовательские сценарии
US-01

Как пользователь без компании, я хочу создать tenant после первого входа, чтобы начать работу в системе.

US-02

Как Primary Owner, я хочу видеть текущий статус доступа tenant, чтобы понимать состояние trial, ограничений или платного доступа.

US-03

Как Primary Owner, я хочу управлять базовыми настройками tenant, чтобы корректно использовать систему.

US-04

Как пользователь, состоящий в нескольких tenant, я хочу переключаться между компаниями, чтобы работать в нужном бизнес-контексте.

US-05

Как SUPPORT_ADMIN, я хочу изменять AccessState tenant в служебных сценариях, чтобы управлять доступом tenant к системе.

7. Функциональные требования
Must Have
FR-01: Система должна позволять пользователю создать tenant после входа в систему.
FR-02: При создании tenant система должна автоматически назначать создателя роли Primary Owner.
FR-03: Система должна обеспечивать строгую логическую изоляцию данных по tenantId.
FR-04: Система должна поддерживать AccessState tenant.
FR-05: Система должна поддерживать следующие состояния AccessState: EARLY_ACCESS, TRIAL_ACTIVE, TRIAL_EXPIRED, ACTIVE_PAID, GRACE_PERIOD, SUSPENDED, CLOSED.
FR-06: Новый tenant должен создаваться в состоянии TRIAL_ACTIVE.
FR-07: Система должна отображать текущий AccessState tenant в интерфейсе Primary Owner.
FR-08: Система должна поддерживать базовые настройки tenant.
FR-09: Система должна хранить налоговую систему tenant как одну из базовых бизнес-настроек компании.
FR-10: Система должна запрещать доступ к данным другого tenant вне активного tenant-контекста.
FR-11: Система должна поддерживать переключение между tenant для пользователей, имеющих доступ к нескольким компаниям.
FR-12: Система должна требовать указание ИНН при создании tenant.
FR-13: Система должна обеспечивать уникальность ИНН в рамках всей системы.
Should Have
FR-14: Система должна позволять Primary Owner изменять название компании.
FR-15: Система должна позволять Primary Owner изменять налоговый режим.
FR-16: Система должна позволять Primary Owner изменять страну, валюту и часовой пояс.
FR-17: Система должна отображать предупреждения о trial expiration, grace period и ограничении доступа.
FR-18: Система должна поддерживать retention window после перевода tenant в CLOSED.
Could Have
FR-20: Система должна поддерживать восстановление tenant из CLOSED в пределах retention window.
FR-21: Система должна поддерживать несколько юрлиц в рамках одного аккаунта по отдельной коммерческой модели.
FR-22: Система должна поддерживать архивирование tenant вместо окончательного удаления.
FR-23: Система должна поддерживать изменение ИНН через отдельный подтвержденный flow или support-сценарий.
FR-24: Система должна поддерживать дополнительные бизнес-настройки tenant beyond базовый набор MVP.
8. Логика создания tenant
Пользователь входит в систему.
Если у пользователя нет ни одного tenant, система показывает онбординг.
Пользователь выбирает создание новой компании.
Пользователь заполняет обязательные поля tenant.
Система создает tenant.
Система назначает создателя tenant ролью Primary Owner.
Новый tenant создается в состоянии TRIAL_ACTIVE.
Новый tenant становится активным tenant-контекстом пользователя.

10. AccessState - машина состояний
EARLY_ACCESS → TRIAL_ACTIVE → ACTIVE_PAID
                    ↓               ↓
              TRIAL_EXPIRED    GRACE_PERIOD
                    ↓               ↓
                SUSPENDED ←────────┘
                    ↓
                 CLOSED
Описание состояний
Состояние	Описание
EARLY_ACCESS	Ранний доступ без оплаты по специальному сценарию
TRIAL_ACTIVE	Активен пробный период
TRIAL_EXPIRED	Пробный период завершен, оплата не выполнена
ACTIVE_PAID	Активная оплаченная подписка
GRACE_PERIOD	Оплата просрочена, но доступ временно еще сохранен
SUSPENDED	Доступ к рабочим операциям tenant ограничен
CLOSED	Tenant закрыт, доступ прекращен
11. Бизнес-эффект AccessState
При EARLY_ACCESS tenant может использовать систему на условиях раннего доступа.
При TRIAL_ACTIVE tenant может использовать систему в рамках пробного периода.
При TRIAL_EXPIRED tenant получает уведомление о завершении trial и ограничение в соответствии с коммерческой политикой.
При ACTIVE_PAID tenant имеет полноценный доступ в рамках оплаченного тарифа.
При GRACE_PERIOD tenant сохраняет доступ временно, но получает уведомления о просрочке.
При SUSPENDED tenant не может выполнять основные рабочие операции, но данные tenant сохраняются.
При CLOSED tenant считается закрытым, доступ к нему прекращается, а данные удерживаются по политике retention.

13. Изоляция данных
Все доменные данные в системе должны быть привязаны к tenant.
Пользователь не должен видеть или изменять данные tenant, к которому у него нет membership.
Все запросы и операции должны выполняться в контексте активного tenant.
Переключение tenant должно менять активный контекст работы пользователя.
Tenant isolation обязательна для всех модулей системы.
14. Работа с несколькими tenant
Один пользователь может иметь доступ к нескольким tenant.
Если у пользователя один tenant, он открывается автоматически.
Если у пользователя несколько tenant, по умолчанию открывается last used tenant.
Если last used tenant недоступен, система показывает экран выбора tenant.
Переключение tenant выполняется через интерфейс приложения без повторного логина.
15. Закрытие tenant и хранение данных
Перевод tenant в SUSPENDED не приводит к удалению данных.
Перевод tenant в CLOSED прекращает доступ к tenant.
После перевода tenant в CLOSED данные tenant должны храниться в течение retention window.
По истечении retention window данные tenant могут быть удалены или архивированы в соответствии с политикой платформы.
Самостоятельное закрытие tenant со стороны Primary Owner не входит в MVP.
Рекомендуемый дефолт
retention window после CLOSED - 30 дней
16. Ограничения и правила
Tenant создается только явным действием пользователя после входа.
Один tenant соответствует одному юрлицу.
Один ИНН может принадлежать только одному tenant в системе.
Изоляция tenant обязательна для всех модулей системы.
AccessState tenant не должен смешиваться с user status или membership status.
Изменение ИНН не входит в свободное редактирование tenant на MVP.
17. Что не входит в этот раздел

Чтобы не размывать границы, в этот раздел не входят:

регистрация и логин пользователя
детальная модель ролей и permissions
приглашение сотрудников как самостоятельный модуль
биллинг и тарификация как отдельный модуль
детальная логика trial-плана
юридическая политика удаления данных и backup lifecycle
поддержка нескольких юрлиц внутри одного tenant на MVP
18. Комментарии на будущее
Возможность объединения нескольких юрлиц в одном аккаунте может быть рассмотрена позже как отдельная коммерческая модель.
Передача роли Primary Owner другому пользователю будет определена позже.
Изменение ИНН tenant будет определено позже через отдельный контролируемый сценарий.
Полный набор бизнес-настроек tenant будет вынесен в отдельный раздел Company Settings.
Подробная логика ролей и прав будет вынесена в отдельный раздел RBAC / Membership / Permissions.
Подробная логика billing lifecycle будет вынесена в отдельный раздел Billing and Subscription.

## 6. AccessState — машина состояний

```
EARLY_ACCESS → TRIAL_ACTIVE → ACTIVE_PAID
                    ↓               ↓
              TRIAL_EXPIRED    GRACE_PERIOD
                    ↓               ↓
                SUSPENDED ←────────┘
                    ↓
                 CLOSED
```

| Состояние | Описание |
|----------|---------|
| EARLY_ACCESS | Ранний доступ без оплаты |
| TRIAL_ACTIVE | Триал активен |
| TRIAL_EXPIRED | Триал истёк, не оплатили |
| ACTIVE_PAID | Активная подписка |
| GRACE_PERIOD | Оплата просрочена, но доступ ещё есть |
| SUSPENDED | Доступ заблокирован |
| CLOSED | Аккаунт закрыт |

---

## 7. Бизнес-правила и ограничения

- Один Tenant ≠ один User. Один User может быть в нескольких Tenant.
- Подписка принадлежит Tenant, не User.
- При SUSPENDED — данные не удаляются.
- При CLOSED — данные удаляются через 30 дней.
