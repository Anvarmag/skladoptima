5. Базовые бизнес-правила
Товар в SkladOptima на MVP трактуется как товарная карточка.
Основная идея каталога - централизованная работа с товарами клиента для синхронизации остатков по нескольким каналам.
Один внутренний товар может быть связан с несколькими маркетплейсами и каналами одновременно.
Один внутренний товар может иметь разные внешние идентификаторы на разных маркетплейсах.
Один внутренний товар может иметь несколько баркодов.
Баркод является справочным полем и не используется как основной ключ автосопоставления.
Название товара не используется как основной ключ автосопоставления.
Основным кандидатом для автоматического сопоставления товаров является SKU.
Если система не находит надежного сопоставления, пользователь должен вручную связать карточки.
При повторной синхронизации уже существующие товары не должны дублироваться.
При soft delete товар не удаляется физически и не удаляется на маркетплейсах.
Soft-deleted товар можно восстановить.
Себестоимость не является обязательным полем карточки товара на MVP и выносится в отдельный аналитический контур.
На MVP допускаются три сценария наполнения каталога:
API-синхронизация
ручное создание
импорт Excel

8. Логика наполнения каталога

На MVP поддерживаются три сценария наполнения каталога:

8.1 API-синхронизация

Основной сценарий наполнения каталога.

Система:

подтягивает товары
подтягивает остатки
подтягивает фото
подтягивает связанные данные

При повторной синхронизации:

существующие товары сохраняются
данные по уже связанным товарам обновляются
новые товары подтягиваются
дубли не создаются
8.2 Ручное создание

Пользователь может вручную создать товар в каталоге.

Такой товар может существовать как карточка в системе и позже быть связан с внешним каналом через SKU или ручное сопоставление.

8.3 Импорт Excel

Импорт Excel является дополнительным сценарием наполнения каталога, а не единственным и не основным.

При импорте:

существующие товары обновляются
дубли не создаются
перед подтверждением доступен preview
ошибки и результаты импорта сохраняются в истории
9. Внутренний каталог vs Marketplace Mapping
Внутренняя карточка товара является основной сущностью каталога.
Один внутренний товар может быть связан с несколькими каналами одновременно.
Для каждого внешнего канала товар может иметь отдельный набор внешних идентификаторов.
Автоматическое сопоставление товаров не должно выполняться по названию.
Автоматическое сопоставление товаров не должно выполняться по баркоду как по основному ключу.
Основным кандидатом для автоматического сопоставления является SKU.
Если система находит надежное совпадение по SKU, товары могут быть автоматически объединены в одну внутреннюю карточку.
Если система не находит надежного сопоставления, пользователь должен вручную связать товары через интерфейс сопоставления.
Для ручного сопоставления система должна показывать товары, которые не были автоматически связаны.

12. Логика удаления товара
Удаление товара в SkladOptima выполняется только как soft delete.
Товар не удаляется физически из истории системы.
Товар не удаляется на маркетплейсах.
Soft delete влияет только на состояние товара внутри SkladOptima.
Soft-deleted товар должен быть скрыт из основного активного каталога.
Soft-deleted товар должен быть доступен через фильтр удаленных товаров.
Soft-deleted товар можно восстановить.
13. Категории, бренд и справочные поля
Категории
На MVP категории поддерживаются как справочное поле для товаров.
Категории используются для фильтрации и удобства навигации по каталогу и остаткам.
Категории могут быть получены из внешних источников или использоваться как внутреннее поле системы.
Бренд
Бренд поддерживается как отдельное справочное поле и отображаемый столбец.
Бренд используется как фильтр и как часть удобного представления товара в каталоге и остатках.
Баркод
Баркод отображается как справочная информация по товару.
Баркод не используется как основной ключ автосопоставления.
14. Фото товара
На MVP у товара поддерживается одно главное фото.
Фото используется для визуального удобства пользователя.
Фото может быть подтянуто автоматически из внешнего канала.
Фото может быть заменено вручную в системе.
Должны поддерживаться ограничения по формату и размеру файла.
Рекомендуемые дефолты
поддерживаемые форматы: JPG, JPEG, PNG, WEBP
максимальный размер файла: до 10 МБ

16. Импорт и история действий
Все импорты должны сохраняться в истории.
Ошибки импорта должны сохраняться в истории.
Действия по товарам должны сохраняться в общей истории системы.
Пользователь должен иметь отдельную вкладку истории, где может видеть изменения и действия в системе.
В истории должны фиксироваться как минимум:
создание товара
редактирование товара
soft delete
восстановление товара
загрузка или замена фото
запуск импорта
завершение импорта
ошибки импорта
создание или изменение marketplace mapping
17. Ограничения и правила
Один баркод не может принадлежать нескольким SKU.
Один товар может иметь несколько баркодов.
Один товар может быть связан с несколькими маркетплейсами.
SKU является основным внутренним ключом товара в системе.
Название и баркод не используются как основной ключ для автоматического связывания товаров.
При импорте существующие товары обновляются, а не дублируются.
Каталог товаров не должен удалять карточки на маркетплейсах.
Модель каталога на MVP не должна зависеть только от одного маркетплейса.
Каталог должен быть пригоден для дальнейшей аналитики и unit-экономики.
