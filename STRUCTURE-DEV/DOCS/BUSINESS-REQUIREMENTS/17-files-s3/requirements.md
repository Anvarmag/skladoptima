1. Описание раздела

Раздел определяет:

загрузку и хранение медиафайлов товаров
связь медиа с товарной карточкой
безопасную выдачу файлов пользователям
tenant-изоляцию файлов
замену и удаление файлов
логирование действий с медиа

На MVP модуль используется для хранения главного фото товара и не является полноценной медиа-платформой или редактором изображений.

2. Бизнес-цели
Обеспечить надежное и масштабируемое хранение фото товаров.
Исключить хранение пользовательских файлов на локальном диске приложения.
Обеспечить безопасную выдачу файлов с tenant-изоляцией.
Обеспечить удобную работу с главным фото товара в каталоге.
Поддержать дальнейшее масштабирование медиа-хранилища без переработки архитектуры.
3. Базовые бизнес-правила
На MVP модуль хранит только фото товаров.
Для товара на MVP поддерживается одно главное фото.
Галерея из нескольких изображений на MVP не поддерживается.
Файлы должны храниться в объектном хранилище, а не на локальном диске приложения.
Файлы должны быть изолированы по tenantId.
Пользовательский ввод не должен напрямую использоваться в object key.
При soft delete товара фото не должно удаляться физически сразу.
Фото soft-deleted товара скрывается вместе с товаром.
Доступ к файлам должен быть защищенным.
Межтенантный доступ к файлам должен быть запрещен.
Замена фото должна поддерживаться без нарушения целостности карточки товара.
Действия с медиа должны логироваться в аудит.
4. Пользовательские сценарии
US-01

Как Manager, я хочу загрузить главное фото товара, чтобы оно отображалось в каталоге.

US-02

Как пользователь, я хочу видеть фото товара в каталоге и других разделах, чтобы быстрее ориентироваться по ассортименту.

US-03

Как Admin, я хочу заменить фото товара, чтобы обновить карточку без удаления самого товара.

US-04

Как система, я хочу хранить фото безопасно и изолированно по tenant, чтобы данные разных компаний не пересекались.

5. Основные сущности
5.1 Media File

Файл, загруженный в объектное хранилище и связанный с сущностью товара.

5.2 Product Main Image

Главное изображение товарной карточки.

5.3 File Metadata

Метаданные файла, хранимые в БД.

6. Функциональные требования
Must Have
FR-01: Система должна поддерживать загрузку главного фото товара в объектное хранилище.
FR-02: Система должна использовать объектное хранилище вместо локального диска приложения.
FR-03: Система должна поддерживать хранение только одного главного фото на товар на MVP.
FR-04: Система должна генерировать object key без прямого использования пользовательского ввода.
FR-05: Система должна хранить в БД object key и метаданные файла.
FR-06: Система должна поддерживать безопасную выдачу файлов.
FR-07: Система должна обеспечивать tenant-изоляцию файлов.
FR-08: Система должна поддерживать замену главного фото товара.
FR-09: Система не должна удалять файл физически при soft delete товара.
FR-10: Система должна скрывать фото soft-deleted товара вместе с самим товаром.
FR-11: Система должна валидировать формат файла при загрузке.
FR-12: Система должна валидировать размер файла при загрузке.
FR-13: Система должна логировать загрузку, замену и удаление/cleanup файлов в аудит.

7. Поддерживаемые файлы на MVP

На MVP поддерживаются только фото товаров.

Не поддерживаются на MVP
галерея товара
документы
вложения других типов
универсальное файловое хранилище для всех сущностей
image moderation
встроенный редактор изображений
8. Ограничения по файлам
Поддерживаемые форматы
JPG
JPEG
PNG
WEBP
Ограничение по размеру
максимальный размер файла: 10 МБ

Примечание: на MVP достаточно ограничения по размеру файла. Более сложные ограничения по пикселям и advanced validation можно добавить позже.

9. Выдача файлов и безопасность
Выдача файлов должна быть защищенной.
На MVP рекомендуется использовать временные или иные защищенные ссылки доступа.
Прямой небезопасный публичный доступ к файлам не является обязательной моделью MVP.
Файл одного tenant не должен быть доступен другому tenant.
Даже если один пользователь связан с несколькими tenant, данные tenant должны оставаться изолированными.
10. Что хранится в БД

В БД по файлу на MVP рекомендуется хранить как минимум:

object key
mime type
size
original filename
uploadedAt
uploadedBy

При необходимости later-ready модель может быть расширена дополнительными метаданными.

11. Замена и удаление файлов
Замена фото
Пользователь может заменить главное фото товара.
При замене новое фото становится активным фото карточки.
Старый файл не должен ломать отображение товара в момент замены.
Soft delete товара
Если товар soft-deleted, фото физически не удаляется сразу.
Фото скрывается вместе с товаром.
При восстановлении товара фото должно оставаться доступным.
Физическое удаление файла

Файл может быть реально удален:

при hard purge
при cleanup замененных файлов
при cleanup orphaned media
по lifecycle policy позже

Для MVP не требуется агрессивное мгновенное удаление всех старых файлов.

Кто может загружать и заменять фото
Primary Owner
Admin
Manager
Кто только просматривает
пользователи, у которых есть доступ к просмотру товара по правилам каталога и ролей
Кто не загружает
Staff
13. Аудит

Система должна логировать как минимум:

загрузку фото
замену фото
удаление / cleanup файла
действия системы, связанные с очисткой или изменением состояния файла, если это реализовано
14. Оптимизация изображений
На MVP допускается базовая оптимизация изображений на уровне delivery pipeline / CDN / object storage integration.
На MVP не требуется сложный собственный image-processing pipeline.
Система должна быть готова к тому, что изображения могут отдаваться в оптимизированном виде через инфраструктурный слой.
15. Ограничения и правила
На MVP модуль не должен разрастаться до полноценного DAM/media-management решения.
Главное фото товара - достаточная модель для первого релиза.
Tenant isolation для файлов обязательна.
Soft delete товара не должен уничтожать связанное медиа немедленно.
Выдача файлов должна быть безопасной.
Локальный диск приложения не должен быть основным хранилищем пользовательских медиа.
16. Что не входит в этот раздел

Чтобы не размывать границы, в этот раздел не входят:

несколько фото на товар
документы и произвольные вложения
image moderation
встроенный редактор изображений
AI image enrichment
advanced optimization pipeline
CDN transformations как отдельный продуктовый модуль
расширенный lifecycle management всех типов файлов
17. Комментарии на будущее
В будущем можно добавить галерею изображений товара.
В будущем можно добавить документы и другие типы файлов.
В будущем можно добавить встроенный редактор изображений.
В будущем можно добавить более сложную оптимизацию, ресайз и lifecycle policy.
В будущем можно добавить более универсальный media layer для разных сущностей системы.
