1. Описание раздела

Раздел определяет:

автоматическую синхронизацию данных между SkladOptima и маркетплейсами
pull и push сценарии синхронизации
обработку остатков и заказов
фоновое выполнение синхронизации
ручной запуск синхронизации из интерфейса
историю sync runs
отображение статусов, ошибок и проблемных подключений
защиту от конфликтов и циклических обновлений

На MVP модуль синхронизации обеспечивает регулярный обмен данными по остаткам и заказам между SkladOptima и подключенными marketplace accounts.

2. Целевая аудитория
Роль	Действия доступны
Primary Owner	Просмотр статусов, ручной запуск синхронизации, просмотр логов
Admin	Просмотр статусов, ручной запуск синхронизации, просмотр логов
Manager	Просмотр статуса синхронизации
Staff	Не управляет синхронизацией
3. Бизнес-цели
Обеспечить регулярную и стабильную синхронизацию остатков и заказов.
Поддержать модель, в которой SkladOptima является master-системой управляемого остатка.
Исключить ручной контроль большей части синхронизационных сценариев.
Обеспечить прозрачность состояния синхронизации для пользователя.
Снизить риск расхождений между SkladOptima и маркетплейсами.
Обеспечить диагностику ошибок и проблемных подключений.
4. Основные сущности
4.1 Marketplace Sync Run

Отдельная запись о запуске синхронизации по конкретному marketplace account.

4.2 Pull Sync

Получение данных из маркетплейса в SkladOptima.

4.3 Push Sync

Отправка данных из SkladOptima в маркетплейс.

4.4 Order Sync

Обработка заказов и связанных событий, влияющих на резерв, списание и статус остатков.

4.5 Sync Status

Текущее состояние синхронизации по marketplace account или каналу.

4.6 Sync Error

Ошибка синхронизации, подключения, данных или прав доступа.

5. Базовые бизнес-правила
На MVP синхронизируются:
остатки
заказы
SkladOptima должна как получать данные из маркетплейсов, так и отправлять обновленные остатки в маркетплейсы.
Частота синхронизации не фиксируется как жесткое бизнес-значение и должна определяться ограничениями API маркетплейсов и правилами интеграции.
Для FBS контура поддерживаются pull и push сценарии.
Для FBO контура поддерживается только pull / display, без push в маркетплейс.
FBO события не должны влиять на мастер-остаток FBS.
При ручной корректировке остатка push в каналы должен выполняться сразу.
Если для товара и канала включен lock / override, auto push в этот канал не должен выполняться.
Устаревшее внешнее событие после ручной корректировки должно фиксироваться как конфликт.
Ошибка одного marketplace account не должна останавливать синхронизацию по другим подключениям tenant.
Синхронизация должна выполняться в фоне и не должна блокировать интерфейс пользователя.
6. Пользовательские сценарии
US-01

Как Primary Owner, я хочу видеть время последней успешной синхронизации, чтобы понимать актуальность данных.

US-02

Как Admin, я хочу вручную запустить синхронизацию, если нужно срочно обновить данные.

US-03

Как система, я должна автоматически синхронизировать остатки и заказы по подключенным marketplace accounts.

US-04

Как пользователь, я хочу видеть ошибку синхронизации в интерфейсе, чтобы быстро понять, что интеграция требует внимания.

US-05

Как пользователь, я хочу видеть проблемные подключения отдельно и понимать, какой канал не синхронизируется.

FR-07: При отмене заказа система должна снимать резерв и возвращать quantity в available stock.
FR-08: При финальном исполнении заказа система должна уменьшать on_hand / FBS stock.
FR-09: Система должна поддерживать ручной запуск синхронизации из UI.
FR-10: Система должна учитывать channel lock / override при выполнении push.
FR-11: Система должна обнаруживать конфликт устаревших внешних событий после ручной корректировки.
FR-12: Система должна сохранять историю sync runs в БД.
FR-13: Система должна отображать статус синхронизации по каждому marketplace account.
FR-14: Система должна отображать lastSyncAt по каждому marketplace account.
FR-15: Система должна отображать ошибки синхронизации в UI и логах.
FR-16: Система должна поддерживать дедупликацию заказов по внешнему идентификатору заказа.
FR-17: Система должна выполняться в фоне и не блокировать UI.

8. Типы синхронизации на MVP
8.1 Pull Sync

Получение данных из маркетплейса в SkladOptima:

заказы

8.2 Push Sync

Передача управляемых остатков из SkladOptima в маркетплейсы.

8.3 Order Sync

Обработка заказов:

создание резерва

финальное списание при исполнении
9. Логика синхронизации остатков
Для FBS
pull поддерживается
push поддерживается
события продаж из маркетплейсов влияют на master stock FBS
Для FBO
pull поддерживается
push не поддерживается
FBO используется только для отображения и анализа
FBO не влияет на master stock FBS
10. Логика заказов в Sync
При поступлении заказа
available stock уменьшается
reserved stock увеличивается
При отмене заказа
reserved stock уменьшается
available stock восстанавливается
При финальном исполнении
on_hand / FBS stock уменьшается
reserved stock уменьшается
Возвраты
возврат может фиксироваться как событие
возврат не увеличивает остаток автоматически
увеличение остатка после возврата выполняется только вручную через Inventory корректировку
11. Channel Lock / Override в Sync
Если для товара и канала включен lock / override, auto push в этот канал не выполняется.
Pull из канала может продолжать работать по общим правилам.
Lock / override должен учитываться sync-логикой как более высокий приоритет, чем стандартный auto push.
В интерфейсе должно быть видно, что канал находится в режиме lock / override.
12. Конфликты и устаревшие события
Если после ручной корректировки приходит устаревшее внешнее событие, система не должна применять его безусловно.
Такое событие должно быть зафиксировано как конфликт.
Конфликт должен быть записан в историю.
Пользователь должен видеть предупреждение о конфликте в UI.
Конфликт одного товара/события не должен останавливать всю синхронизацию tenant.
13. История Sync Runs

Для каждого запуска синхронизации система должна сохранять запись в БД.

MarketplaceSyncRun должен включать как минимум:
marketplace
marketplace account
startedAt
finishedAt
status
lastSyncAt
количество обработанных сущностей
количество ошибок
текст или код ошибки при наличии
Рекомендуемые статусы sync run
success
partial_success
failed

В UI по каждому marketplace account на MVP рекомендуется показывать:

marketplace
label / название подключения
статус подключения
статус синхронизации
lastSyncAt
warning / error state, если есть
Права доступа

Manager видит только статус
Staff не управляет синхронизацией
15. Ошибки и уведомления
Ошибки синхронизации должны логироваться.
Ошибки синхронизации должны отображаться в UI.
Ошибки одного marketplace account не должны ломать синхронизацию других account.
Пользователь должен видеть, какой именно канал / account находится в ошибочном состоянии.
На MVP рекомендуется показывать:
invalid credentials
sync failed
conflict detected
needs reconnect
inactive channel
16. Выполнение синхронизации
Синхронизация должна выполняться в фоне.
Синхронизация не должна блокировать пользовательский интерфейс.
Архитектурно выполнение может быть реализовано через отдельный worker / background processor.
Конкретная техническая реализация не является частью этого BRD, но фоновая модель обязательна как поведение системы.
17. Ограничения и правила
Частота синхронизации должна подстраиваться под ограничения API и rate limits маркетплейсов.
Нельзя фиксировать единую жесткую частоту для всех сценариев как универсальное правило бизнеса.
Push и pull логика должна учитывать channel locks и конфликты.
Заказы должны обрабатываться идемпотентно.
Повторное внешнее событие по уже обработанному заказу не должно дублировать бизнес-эффект.
История sync runs обязательна для диагностики и поддержки.
Технический sync log и история движений остатков должны быть разделены.
18. Что не входит в этот раздел

Чтобы не размывать границы, в этот раздел не входят:

детальная логика inventory как отдельный модуль
детальная логика marketplace credentials и подключений как отдельный модуль
event-driven streaming и сложная webhook-first архитектура
авторазбор конфликтов
сложные per-account sync policies
расширенная rate-limit orchestration
сложный reconciliation engine между маркетплейсами
сложная логика возвратов beyond basic logging
ручное управление FBO остатками
19. Комментарии на будущее
В будущем можно разделить частоту sync по типам данных.
В будущем можно добавить более гибкие политики sync per account.
В будущем можно расширить обработку конфликтов.
В будущем можно добавить более глубокую диагностику причин sync failures.
В будущем можно поддержать более событийную модель вместо базового polling-подхода.
В будущем можно расширить sync не только на остатки и заказы, но и на другие сущности.
