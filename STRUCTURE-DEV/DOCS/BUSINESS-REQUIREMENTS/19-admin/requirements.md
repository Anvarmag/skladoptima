1. Описание раздела

Раздел определяет:

внутреннюю административную панель команды SkladOptima
просмотр и сопровождение tenant
служебные действия поддержки
просмотр ключевых состояний tenant, подписки и интеграций
безопасные support-операции с обязательным аудитом
внутренние заметки поддержки

На MVP Admin-панель предназначена для команды SkladOptima и не является частью пользовательского кабинета tenant.

2. Целевая аудитория
Роль	Действия доступны
SUPPORT_ADMIN	Полный доступ к admin-панели и support-операциям
Внутренняя команда SkladOptima	Доступ по внутренним правилам и ролям платформы
3. Бизнес-цели
Дать команде SkladOptima единый инструмент сопровождения tenant.
Ускорить диагностику проблем пользователей и интеграций.
Поддержать безопасные ручные действия поддержки без прямого вмешательства в пользовательские пароли и секреты.
Обеспечить прозрачность всех support-операций через аудит.
Создать foundation для масштабируемой внутренней операционной панели.
4. Базовые бизнес-правила
Admin-панель является внутренним инструментом команды SkladOptima.
Доступ к admin-панели имеют только служебные роли платформы.
High-risk действия поддержки должны логироваться особенно строго.
Support не должен видеть пароли пользователей.
Support не должен использовать небезопасные способы “восстановления доступа”.
Support может инициировать безопасный password reset flow.
Support может менять состояние tenant только через явные support-операции.
Support-операции должны сопровождаться reason/comment.
Impersonation не входит в MVP.
Полноценный встроенный тикетинг не входит в MVP.
Для сопровождения tenant на MVP допускается использование внешних каналов поддержки плюс internal notes внутри admin-панели.
5. Основные сущности
5.1 Tenant Directory

Список tenant в системе с поиском, фильтрацией и краткими статусами.

5.2 Tenant Admin View

Карточка tenant с обзором состояния tenant, подписки, команды, интеграций и последних проблем.

5.3 Support Action

Ручное служебное действие поддержки по tenant или пользователю.

5.4 Internal Support Note

Внутренняя заметка команды поддержки по tenant.

5.5 Support Audit Entry

Аудит-запись, фиксирующая служебное действие, причину и контекст.

6. Пользовательские сценарии
US-01

Как SUPPORT_ADMIN, я хочу видеть список всех tenant и их состояние, чтобы быстро находить проблемные аккаунты.

US-02

Как SUPPORT_ADMIN, я хочу открыть карточку tenant и увидеть подписку, интеграции и недавние ошибки, чтобы быстро понять контекст.

US-03

Как SUPPORT_ADMIN, я хочу вручную продлить trial tenant, если это согласовано по бизнес-правилам.

US-04

Как SUPPORT_ADMIN, я хочу изменить access state tenant в разрешенных support-сценариях, чтобы восстановить доступ или исправить ошибочную блокировку.

US-05

Как SUPPORT_ADMIN, я хочу выдать special free access, если tenant относится к партнерскому, внутреннему или исключительному сценарию.

US-06

Как SUPPORT_ADMIN, я хочу инициировать безопасный reset password flow, чтобы помочь пользователю восстановить доступ.

US-07

Как SUPPORT_ADMIN, я хочу оставить внутреннюю заметку по tenant, чтобы команда поддержки видела контекст и историю сопровождения.

11. Support actions и безопасность
Разрешенные действия на MVP
продление trial
изменение access state
назначение special free access
инициирование password reset flow
Ограничения
Support не видит пароль пользователя.
Support не получает доступ к секретам через небезопасный shortcut.
Support не должен входить как пользователь на MVP.
Support actions должны фиксироваться в аудите.
Reason / Comment

Для high-risk support actions обязателен комментарий, например:

продление trial по согласованию
исправление ошибочной блокировки
партнерский special access
ручная помощь клиенту
12. Internal Support Notes
Support должен иметь возможность оставлять внутренние заметки по tenant.
Internal support notes видны только внутренней команде.
Notes используются для контекста сопровождения, handoff между support-сотрудниками и служебной истории tenant.
Notes не заменяют полноценный тикетинг, но покрывают минимальную operational потребность MVP.
Примеры notes
клиенту продлили trial на 7 дней
tenant ждет перевыпуск Ozon ключа
договоренность с sales о special free access
пользователь обращался по проблеме доступа
13. Аудит support-действий
Все support actions должны логироваться.
Audit support actions должен включать:
кто сделал действие
над каким tenant
какое действие выполнено
когда выполнено
reason/comment
Support audit должен быть доступен для внутренней диагностики и контроля.
14. Права доступа
SUPPORT_ADMIN

Имеет доступ к admin-панели и support actions в пределах своей служебной роли.

Пользовательские роли tenant
Primary Owner
Admin
Manager
Staff

Не имеют доступа к internal admin-панели платформы.

15. Ограничения и правила
Admin-панель не должна превращаться в неограниченный “god mode” без следов.
High-risk операции должны быть объяснимыми и аудируемыми.
Support не должен выполнять небезопасные действия с пользовательскими учетными данными.
Impersonation не входит в MVP.
Полноценный встроенный тикетинг не входит в MVP.
External support channels могут использоваться параллельно с admin-панелью.
Internal notes являются минимальным operational replacement для части тикетинговых сценариев MVP.
16. Что не входит в этот раздел

Чтобы не размывать границы, в этот раздел не входят:

impersonation / login as user
просмотр или восстановление пароля в открытом виде
полноценный встроенный тикетинг
омниканальная support CRM
сложные approval workflows для support actions
расширенная инфосек-панель расследований
DevOps/infra monitoring console
