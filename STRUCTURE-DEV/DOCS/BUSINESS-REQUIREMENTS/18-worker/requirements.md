1. Описание раздела

Раздел определяет:

выполнение тяжелых и отложенных задач вне основного HTTP API
фоновую обработку синхронизации, уведомлений и служебных задач
scheduled jobs по расписанию
retry-логику и обработку failed jobs
наблюдаемость и статус фоновых задач
устойчивость системы к сбоям и рестартам

На MVP модуль фоновых задач обеспечивает, чтобы длительные операции не блокировали пользовательский интерфейс и основное API, а выполнялись надежно и прозрачно в фоне.

2. Бизнес-цели
Исключить блокировку основного API тяжелыми операциями.
Обеспечить надежное выполнение синхронизации, уведомлений и периодических задач.
Обеспечить автоматическое восстановление и повторные попытки для временно неуспешных задач.
Дать пользователю понятный статус выполнения критичных фоновых процессов.
Дать support/admin возможность видеть проблемные фоновые задачи и локализовать сбои.
3. Базовые бизнес-правила
Тяжелые и длительные операции не должны выполняться синхронно в пользовательском HTTP-запросе.
Фоновые задачи должны выполняться отдельно от основного API.
Ошибка одной фоновой задачи не должна останавливать весь процесс обработки задач.
Фоновые задачи должны поддерживать автоматические повторные попытки для временных ошибок.
Повторные попытки должны выполняться с увеличивающимся интервалом.
Неуспешные после всех попыток задачи не должны теряться молча.
Критичные фоновые процессы должны иметь понятный статус в UI.
Система должна быть устойчива к рестартам и не должна терять задачи без явного статуса.
Фоновые задачи и их технический мониторинг являются отдельным инфраструктурным слоем, но их бизнесовые статусы должны быть видимы пользователям по роли.
4. Основные сущности
4.1 Background Job

Фоновая задача, поставленная системой в очередь на выполнение.

4.2 Scheduled Job

Фоновая задача, запускаемая автоматически по расписанию.

4.3 Retry Policy

Правила повторной попытки выполнения задачи при временной ошибке.

4.4 Failed Job

Задача, которая не выполнилась даже после всех повторных попыток.

4.5 Job Status

Бизнесовый или технический статус фоновой задачи.

5. Пользовательские сценарии
US-01

Как система, синхронизация маркетплейсов не должна блокировать основное API.

US-02

Как пользователь, я хочу видеть статус синхронизации, чтобы понимать, выполняется ли она сейчас и завершилась ли успешно.

US-03

Как система, я хочу автоматически повторять временно неуспешные задачи, чтобы не требовать ручного вмешательства в каждом временном сбое.

US-04

Как Primary Owner или Admin, я хочу видеть, что фоновая задача завершилась ошибкой, чтобы понять, что системе требуется внимание.

US-05

Как SUPPORT_ADMIN, я хочу видеть проблемные фоновые задачи и failed jobs, чтобы локализовать и решать инциденты.

6. Какие задачи выполняются в фоне на MVP

На MVP в фоне должны выполняться как минимум:

синхронизация маркетплейсов
pull/push остатков
обработка заказов
отправка email
отправка Telegram уведомлений
отправка уведомлений в Max
ежедневная сводка по low stock
billing reminders
cleanup jobs по замененным файлам и служебному медиа-мусору
периодическое обновление аналитики, если это требуется продуктовой логикой
7. Какие задачи запускаются по расписанию

На MVP по расписанию должны запускаться:

регулярная синхронизация маркетплейсов
ежедневная сводка low stock
уведомления о trial / subscription / billing reminders
cleanup jobs
периодический пересчет или обновление аналитики, если включено системой

Примечание: точное расписание и интервалы определяются бизнес-логикой, rate limits внешних API и настройками системы.

FR-03: Система должна поддерживать фоновые задачи для email.

FR-09: Система должна поддерживать мониторинг состояния фоновых задач.
FR-10: Система должна отображать статус важных фоновых задач в UI.

FR-14: Система должна быть устойчива к рестартам и не должна терять задачи без статуса.

9. Retry и failed jobs
Для временных ошибок система должна выполнять автоматические повторные попытки.
Повторные попытки должны выполняться по модели increasing backoff.
Если задача не выполнилась после допустимого количества попыток, она должна перейти в failed state.
Failed job не должна теряться молча.
Failed job должна быть видна в мониторинге и доступна для расследования.
Повторные попытки должны использоваться только для ошибок, которые потенциально можно восстановить автоматически.
10. Статусы фоновых задач

Для бизнесового и операционного UI на MVP рекомендуется поддерживать статусы:

queued
in_progress
retrying
success
failed
Примеры
sync queued
sync in progress
sync retrying
sync success
sync failed
11. Отображение в UI
Для Primary Owner / Admin

В UI должны отображаться понятные бизнесовые статусы критичных фоновых процессов tenant, например:

синхронизация запущена
синхронизация выполняется
синхронизация завершена
синхронизация завершилась ошибкой
выполняется повторная попытка
Для SUPPORT_ADMIN

Должен быть доступен более глубокий просмотр состояния фоновых задач и failed jobs для диагностики и поддержки.

Для обычного пользователя

Глубокая техническая информация о job runner/queue не требуется.

12. Права и уровни видимости
Primary Owner / Admin

Видят бизнесовые статусы фоновых задач своего tenant.

Manager

Может видеть ограниченные бизнесовые статусы там, где это связано с его operational flow, например sync status.

SUPPORT_ADMIN

Видит расширенную информацию по проблемным задачам и failed jobs.

DevOps / техническая команда

Глубокий технический мониторинг очередей и worker-инфраструктуры не является частью обычного продуктового UI.

13. Устойчивость и восстановление
Фоновые задачи не должны теряться при рестарте worker-процесса или приложения.
Система должна корректно восстанавливаться после рестартов.
Задачи, находившиеся в очереди или в безопасном промежуточном состоянии, должны продолжать жизненный цикл по правилам системы.
Фоновые процессы должны корректно завершать работу при штатной остановке системы, насколько это возможно в рамках инфраструктуры.
14. Наблюдаемость и мониторинг

На MVP система должна поддерживать базовую наблюдаемость по фоновым задачам:

какие задачи в очереди
какие выполняются
какие упали
какие повторяются
какие завершились успешно

Глубина мониторинга для внутренних технических команд может быть выше, чем для продуктового UI.

15. Технологические замечания
Система должна использовать очередь фоновых задач и выделенный механизм выполнения background jobs.
Конкретный выбор библиотеки, брокера и технологической реализации не является частью бизнес-требований.
Архитектурно система должна поддерживать:
очередь задач
retries
delayed jobs
monitoring
failed jobs visibility

Конкретная технологическая связка фиксируется в архитектурном решении, а не в этом BRD.

16. Ограничения и правила
Фоновый слой не должен дублировать основную бизнес-логику, а должен надежно ее исполнять.
Пользователь не должен ждать тяжелые операции в синхронном API-запросе.
UI должен отображать только тот уровень информации, который нужен роли.
Система должна различать scheduled jobs и event-driven queued jobs.
Сбой одной задачи не должен приводить к каскадной остановке остальных задач.
17. Что не входит в этот раздел

Чтобы не размывать границы, в этот раздел не входят:

детальная реализация конкретной очереди
выбор конкретной библиотеки worker/queue
низкоуровневый DevOps monitoring stack
детальная трассировка инфраструктурных проблем
ручное управление всеми очередями из пользовательского UI
сложная оркестрация workflow уровня BPM engine
18. Комментарии на будущее
В будущем можно добавить отдельные очереди по типам задач.
В будущем можно добавить приоритизацию задач.
В будущем можно добавить более гибкие политики retries.
В будущем можно расширить мониторинг фоновых процессов.
В будущем можно добавить более сложные workflow и orchestration сценарии.
