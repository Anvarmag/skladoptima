---

## 1. Общие принципы

- **Простота важнее элегантности.** Три понятные строки лучше одной умной абстракции.
- **Не делаем то, о чём не просили.** Баг-фикс — это только баг-фикс. Не рефакторим соседний код.
- **Сначала требования, потом код.** Ни одна фича не начинается без заполненного `requirements.md`.
- **Документация обновляется вместе с кодом** — в том же PR/коммите, не потом.

## 3. База данных

### Правила Prisma

- Только `prisma migrate dev` — никаких `$executeRawUnsafe` / `ALTER TABLE` в коде.
- Имя каждой миграции: `<что_изменилось>` (например: `add_invitation_model`).
- Перед prod: `prisma migrate deploy`.
- Индексы добавляются **вместе** с моделью, не отдельной миграцией.
- Prisma Client используется **только** внутри соответствующего модуля через его сервис. Прямой доступ из других модулей запрещён.

### Naming конвенции

- Таблицы: `PascalCase` (как в Prisma schema).
- Поля: `camelCase`.
- Enum значения: `UPPER_SNAKE_CASE`.
- Индексы по `tenantId` обязательны во всех tenant-scoped таблицах.

### Запрещено

- `onModuleInit()` для структурных изменений БД.
- Singleton-паттерны типа `id = '1'`.
- Хранение файлов на диске app-сервера.

## 4. Backend (NestJS)

### Структура модуля

```
modules/<name>/
  <name>.module.ts
  <name>.controller.ts
  <name>.service.ts
  dto/
    create-<name>.dto.ts
    update-<name>.dto.ts
```

### Правила

- Один модуль — одна ответственность.
- DTO валидируются через `class-validator`.
- Все роуты защищены `JwtAuthGuard` по умолчанию. Публичные — явный `@Public()` декоратор.
- Tenant isolation: каждый запрос обязан проверять `tenantId` из JWT.
- JWT содержит: `userId`, `tenantId`, `membershipId`, `role`. Никаких `storeId`.
- Ошибки: `HttpException` с понятными сообщениями. Не раскрывать внутренние детали.
- Логирование: через NestJS Logger, не `console.log`.

### Запрещено

- Прямой доступ к `PrismaService` из контроллера (только через сервис).
- Хардкод tenantId, userId в логике.
- Бизнес-логика в контроллере.

## 5. Frontend (React)

### Структура (цель — FSD)

```
src/
  app/        — провайдеры, роутинг
  pages/      — страницы
  widgets/    — составные блоки UI
  features/   — бизнес-фичи
  entities/   — доменные сущности
  shared/     — ui-kit, utils, api, types
```

### Правила

- Все тексты интерфейса на **русском**.
- Нет прямых fetch/axios вне `shared/api`.
- Состояние — React state / context. Не добавлять Redux/Zustand без обсуждения.
- Формы — нативный React state или React Hook Form (если уже используется).
- Компонент не знает о деталях API — только о пропсах.
- Нет inline-стилей. Только Tailwind классы.

---

## 6. Безопасность

- Никакого XSS: не вставлять `dangerouslySetInnerHTML` без review.
- SQL injection невозможна при использовании Prisma ORM (не raw queries).
- Все API-ключи маркетплейсов хранятся зашифрованными (не plaintext).
- S3 ключи файлов: `{tenantId}/products/{uuid}.{ext}` — никогда не содержат пользовательский ввод напрямую.
- CORS настроен строго: только разрешённые origins.
- Helmet подключён и настроен.

## 7. Workflow спринта

### Жизненный цикл задачи

```
BUSINESS-REQUIREMENTS заполнен
        ↓
ANALYTICS заполнен
        ↓
PLAN-SPRINT-X.md создан (задачи + этапы)
        ↓
TASKS-SPRINT-X/tasks.md заполнен (конкретные dev-задачи)
        ↓
Разработка
        ↓
PLAN-SPRINT-X.md обновлён (чекбоксы + что сделано)
```

### Правила спринта

- Спринт = **2 недели**.
- Не начинать спринт без заполненного PLAN-SPRINT-X.md.
- Не добавлять задачи в середине спринта без обновления PLAN.
- После каждой выполненной задачи: ставим `[x]` в PLAN + добавляем описание "что сделано".
- Задачи из TASKS не удаляются — только перемещаются в следующий спринт с пометкой.

## 8. Git

- Ветка на каждый спринт: `sprint/X-название`.
- Коммиты на русском или английском — но единообразно внутри спринта.
- Формат коммита: `[S1] Краткое описание` где S1 — номер спринта.
- Никаких `--no-verify`, `--force` на `main` без явного разрешения.
- PR в `main` только после review.

## 9. Документация

- Каждый раздел продукта имеет свои файлы: `requirements.md`, `analytics.md`.
- Шаблоны: `_TEMPLATE.md` в каждой папке — не редактировать, только копировать.
- Если требования изменились — обновляем `requirements.md`, не создаём новый файл.
- PLAN.md в корне — технический обзор. Детальные планы — в `STRUCTURE-DEV/DOCS/PLAN/`.
