# Пользовательский путь (User Flow) и Архитектура "Под капотом"

В этом документе описано, как будет выглядеть процесс работы SaaS-платформы с подписками со стороны клиента (Пользователя) и параллельно — со стороны серверной логики (NestJS + Prisma + PostgreSQL).

---

## 1. Регистрация (Создание аккаунта компании)

**👨‍💻 Действия пользователя:**
1. Заходит на сайт по реферальной ссылке (например, `?ref=ref-123`).
2. Вводит Email, Пароль, Название своего магазина (компании).
3. Вводит промокод на скидку: "START20".
4. Нажимает "Зарегистрироваться".

**⚙️ Что происходит под капотом (NestJS + Prisma):**
1. **Frontend (React)** отправляет POST `/auth/register` с телом: `{ email, password, storeName, refCode, promoCode }`.
2. **Backend (NestJS)**:
   - Хеширует пароль через `bcrypt`.
   - Проверяет `promoCode` в таблице `PromoCode`. Если найден и не истек — резервирует его.
   - Ищет пригласившего по `refCode` в таблице `Store` (колонка `referralCode`).
   - Через **Prisma Transaction** (чтобы всё создалось разом или ничего):
     - Создает новый `Store` с названием магазина. Присваивает `referredById` (id пригласившего аккаунта).
     - Создает `User` с привязкой к новому `Store` и устанавливает ему роль: `role: "OWNER"`.
     - Генерирует и возвращает JWT-токен в `httpOnly` cookie.

---

## 2. Покупка подписки (Оплата SaaS)

**👨‍💻 Действия пользователя (OWNER):**
1. Пользователь заходит в раздел "Оплата / Подписка" (ему доступен триал на 3 дня, который создался при регистрации).
2. Выбирает тариф "PRO" (до 10 сотрудников, безлимит товаров).
3. Оплачивает картой.

**⚙️ Что происходит под капотом:**
1. **Frontend** отправляет запрос в `SubscriptionController`.
2. **Backend**:
   - NestJS проверяет по JWT-токену и Guard-у (`@Roles('OWNER')`), что запрос делает именно Владелец, а не Менеджер.
   - Перенаправляет запрос на платежный шлюз (например, ЮKassa / Stripe).
3. **Webhook (Асинхронно)**:
   - Платежный шлюз присылает Webhook на бэкенд об успешной оплате.
   - Бэкенд находит `Store` пользователя и через Prisma обновляет или создает сущность `Subscription`:
     `prisma.subscription.create({ data: { storeId: user.storeId, plan: 'PRO', validUntil: +1 месяц } })`.
   - Если пользователь пришел по реферальной ссылке (`referredById` не пустой), бэкенд начисляет бонус пригласившему магазину:
     `prisma.store.update({ where: { id: referredById }, data: { bonusBalance: { increment: 500 } } })`.

---

## 3. Приглашение сотрудников (Управление ролями)

**👨‍💻 Действия пользователя (OWNER):**
1. Заходит в раздел "Сотрудники".
2. Нажимает "Добавить сотрудника", вводит email: `manager@mail.com`, выбирает роль: `MANAGER`.
3. Система генерирует временный пароль для входа.

**⚙️ Что происходит под капотом:**
1. **Backend**:
   - `RolesGuard` проверяет, что действие делает `OWNER`.
   - `SubscriptionService` проверяет лимиты тарифа (Например: "в тарифе PRO максимум 10 пользователей"). 
   - `prisma.user.create()` создает нового пользователя: привязывает его к тому же `storeId`, задает `role: "MANAGER"`.

---

## 4. Повседневная работа (Изменение остатков товаров)

**👨‍💻 Действия пользователя (MANAGER):**
1. Менеджер авторизуется под `manager@mail.com`.
2. Заходит в "Товары" и вручную списывает 5 штук товара "Футболка".

**⚙️ Что происходит под капотом:**
1. **Backend**:
   - По JWT-токену из cookie сервер понимает: `email: manager@mail...`, `storeId: 123...`, `role: MANAGER`.
   - `ProductService` берет товар.
   - Через Prisma атомарно `decrement` товара: `data: { total: { decrement: 5 } }`.
   - **(Важно!)** `AuditService` делает запись в логи:
     `prisma.auditLog.create({ data: { action: "STOCK_ADJUSTED", actorEmail: user.email, actorRole: user.role, delta: -5, storeId: user.storeId } })`.

---

## 5. Просмотр отчетов (Аудит и безопасность)

**👨‍💻 Действия пользователя (OWNER):**
1. Владелец заходит в раздел "История действий".
2. Видит таблицу: 
   - *14:05 | Списано 5 шт. (Остаток 15) | Менеджер (manager@mail.com)*

**⚙️ Что происходит под капотом:**
1. **Frontend** делает `GET /audit` с пагинацией.
2. **Backend**:
   - Отдает только логи с `storeId === OWNER.storeId` (безопасность — нельзя увидеть данные других компаний).
   - В логах уже зафиксирована роль `actorRole`, поэтому фронтенду легко нарисовать бейджик "Менеджер" или "Админ".

---

### Вывод
Вся логика строится вокруг строгой изоляции данных по `storeId` на уровне базы данных PostgreSQL и проверки прав доступа (Guard-ов) на уровне NestJS. Prisma позволяет связывать данные (`Store` -> `User` -> `AuditLog`) и гибко контролировать транзакции (чтобы деньги не списались, если не обновилась подписка).
