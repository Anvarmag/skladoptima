## 1. Назначение раздела

Раздел `CHANNEL CONTROLS` решает три операционных задачи управления остатками по каналам продаж:

1. **Блокировка синхронизации** — возможность заморозить отправку остатков на конкретный маркетплейс для конкретного товара, чтобы система не перезаписывала вручную выставленное значение.
2. **Фильтр видимости каналов** — возможность скрывать/показывать колонки маркетплейсов в интерфейсе остатков, чтобы не перегружать экран неактуальными данными.
3. **Склейка артикулов** — явный и удобный UI для связывания разных артикулов с разных площадок в один внутренний товар.

## 2. Целевая аудитория и роли

| Роль | Что хочет получить | Основные действия |
|------|--------------------|-------------------|
| OWNER / ADMIN | Контроль над тем, что уходит на каждую площадку | Ставить блокировки, управлять видимостью, делать склейку |
| MANAGER | Операционная работа без лишнего шума | Фильтровать видимость каналов, смотреть связанные артикулы |
| Система (sync) | Уважать пользовательские блокировки | Не перезаписывать заблокированные значения |

## 3. Бизнес-цели

- [ ] BG-01: Дать пользователю контроль над тем, что отправляется на каждый маркетплейс.
- [ ] BG-02: Устранить проблему перезаписи вручную выставленных нулевых остатков при синхронизации.
- [ ] BG-03: Упростить работу с таблицей остатков за счёт скрытия неактуальных каналов.
- [ ] BG-04: Устранить дублирование товаров от разных площадок через удобный интерфейс склейки.

## 4. Scope раздела

### Что входит

- модель данных и API для блокировок синхронизации по каналу (`StockChannelLock`);
- интеграция блокировок в pipeline push_stocks;
- настройки видимости каналов в интерфейсе остатков (на уровне тенанта);
- UI для управления блокировками прямо из таблицы остатков;
- UI для склейки артикулов (связывание внутреннего товара с внешними артикулами);
- audit блокировок и изменений маппинга.

### Что не входит

- изменение алгоритма pull_stocks (данные продолжают приходить, блокируется только push);
- автоматические блокировки без явного действия пользователя;
- полноценный WMS-модуль управления физическими складами;
- мультивалютные overrides.

## 5. Пользовательские сценарии

- **US-01:** Как OWNER, я хочу выставить 0 на ВБ для конкретного товара и заблокировать, чтобы sync не перезаписывал — пока я распродаю FBO.
- **US-02:** Как MANAGER, я хочу скрыть в таблице остатков колонку Ozon, если я сейчас работаю только с ВБ.
- **US-03:** Как OWNER, я хочу явно связать артикул ВБ и артикул Ozon в один товар в системе, чтобы видеть суммарный остаток.
- **US-04:** Как система (sync), я обязана перед отправкой остатка проверить, есть ли блокировка, и если есть — отправить зафиксированное значение вместо реального.
- **US-E01:** Что происходит, если пользователь снял блокировку во время выполнения sync run.
- **US-E02:** Что происходит, если товар удалён, а блокировка осталась.

## 6. Функциональные требования

### Must Have

- [ ] FR-01: Система должна позволять создавать блокировку синхронизации для пары (товар, маркетплейс).
- [ ] FR-02: Система должна поддерживать три типа блокировки: ZERO (всегда 0), FIXED (фиксированное значение), PAUSED (не отправлять).
- [ ] FR-03: Pipeline push_stocks обязан проверять блокировки перед каждой отправкой.
- [ ] FR-04: Система должна позволять снять блокировку с немедленным эффектом для следующего sync run.
- [ ] FR-05: Система должна поддерживать настройки видимости каналов на уровне тенанта.
- [ ] FR-06: Система должна поддерживать явное связывание внешнего артикула (externalProductId) с внутренним товаром через UI.

### Should Have

- [ ] FR-10: Система должна показывать badge/индикатор на заблокированных товарах в таблице остатков.
- [ ] FR-11: Система должна логировать создание и снятие блокировок в audit.
- [ ] FR-12: Система должна поддерживать bulk-снятие блокировок для тенанта.

### Could Have

- [ ] FR-20: Система должна поддерживать авто-подсказки склейки (auto-match по barcode/name similarity).
- [ ] FR-21: Система должна поддерживать временные блокировки с датой истечения.

## 7. Бизнес-правила и ограничения

- BR-01: Блокировка не влияет на pull_stocks — данные с маркетплейса продолжают приходить и отображаться в информационном контуре.
- BR-02: При FIXED блокировке значение не может быть отрицательным.
- BR-03: Склейка (ProductChannelMapping) разрешает только одну связь на пару (маркетплейс, externalProductId) в рамках тенанта.
- BR-04: Настройки видимости — это UI-фильтр, они не влияют на сами данные и синхронизацию.
- BR-05: Блокировка создаётся только для активных маркетплейс-аккаунтов.

## 8. Пользовательские состояния и ошибки

- Состояние: товар заблокирован на маркетплейсе — sync отправляет зафиксированное значение.
- Состояние: товар не связан ни с одним каналом — отображается в разделе «Несвязанные».
- Ошибка: попытка создать блокировку для неактивного аккаунта маркетплейса.
- Ошибка: попытка FIXED блокировки с отрицательным значением.
- Ошибка: конфликт — пытаются связать уже связанный externalProductId с другим товаром.

## 9. Нефункциональные ожидания

| Область | Ожидание |
|---------|----------|
| Надёжность | Блокировки должны применяться атомарно в pipeline push_stocks |
| Производительность | Проверка блокировок не должна добавлять более 10 мс к каждому push item |
| Прозрачность | Пользователь всегда должен видеть, на каком товаре и на каком канале стоит блокировка |
| Аудит | Все создания/снятия блокировок логируются |
