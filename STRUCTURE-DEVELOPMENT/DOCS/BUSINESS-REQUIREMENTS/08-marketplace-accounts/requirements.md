## 1. Назначение раздела

Раздел `MARKETPLACE-ACCOUNTS` управляет подключением внешних кабинетов маркетплейсов к tenant. Он нужен для того, чтобы tenant мог безопасно подключать каналы продаж, видеть статус интеграции и управлять credentials без потери истории.

## 2. Целевая аудитория и роли

| Роль | Что хочет получить | Основные действия |
|------|--------------------|-------------------|
| OWNER | Контроль интеграций tenant | Создание, отключение, проверка подключения |
| ADMIN | Операционная работа с подключениями | Перепроверка статуса, обновление данных |
| MANAGER | Понимание состояния каналов | Просмотр статусов |

## 3. Бизнес-цели

- [ ] BG-01: Упростить подключение маркетплейсов.
- [ ] BG-02: Сделать статус интеграции прозрачным.
- [ ] BG-03: Защитить credentials и историю подключений.

## 4. Scope раздела

### Что входит

- создание и редактирование account;
- хранение credentials и warehouse references;
- проверка валидности подключения;
- статусы подключения;
- безопасное отключение account.

### Что не входит

- полная логика sync;
- финансовая сверка маркетплейсов;
- многопровайдерный integration marketplace beyond agreed list.

## 5. Пользовательские сценарии

- **US-01:** Как OWNER, я хочу подключить WB или Ozon.
- **US-02:** Как пользователь, я хочу видеть валидно ли подключение.
- **US-03:** Как пользователь, я хочу отключить account без потери истории.

## 6. Функциональные требования

### Must Have

- [ ] FR-01: Система должна поддерживать создание marketplace account.
- [ ] FR-02: Система должна хранить статусы подключения.
- [ ] FR-03: Система должна позволять проверить credentials.
- [ ] FR-04: Система должна позволять отключить account без удаления исторических данных.

### Should Have

- [ ] FR-10: Система должна хранить last sync status/error на account.

### Could Have

- [ ] FR-20: Система должна поддерживать несколько account одного маркетплейса.

### Planned (следующие итерации)

- [ ] FR-30: **Dual-token WB** — WB-подключение должно поддерживать два отдельных токена:
  - `apiToken` (обязательный) — операционный токен с правами **Маркетплейс (чтение и запись)**; используется только для управления остатками FBS и заказами
  - `analyticsToken` (необязательный) — аналитический токен с правами **Статистика + Контент (только чтение)**; используется для FBO остатков, карточек товаров, финансовых отчётов
  - Если `analyticsToken` не задан — fallback на `apiToken` (обратная совместимость)
  - Форма создания WB-подключения должна визуально разделять два токена с пояснением прав каждого
  - См. `TASK_MARKETPLACE_ACCOUNTS_8`

- [ ] FR-31: **Подтягивание финансовых данных WB** — при наличии `analyticsToken` система должна автоматически получать детализированный финансовый отчёт WB (`/api/v5/supplier/reportDetailByPeriod`) и заполнять `marketplaceFees` для юнит-экономики без ручного ввода
  - См. `TASK_ANALYTICS_8`

## 7. Бизнес-правила и ограничения

- BR-01: Секреты подключения не должны отображаться в открытом виде.
- BR-02: Отключение account не должно уничтожать исторические данные tenant.
- BR-03: Статус account должен быть понятен пользователю без чтения логов.
- BR-04: Операционный и аналитический токены WB имеют разные уровни доступа — смешивать их в одном поле запрещено; разделение в UI обязательно.

## 8. Пользовательские состояния и ошибки

- Состояние: connected.
- Состояние: invalid credentials.
- Состояние: needs reconnect.
- Состояние: inactive.
- Ошибка: credentials невалидны.
