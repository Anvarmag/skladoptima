## 1. Назначение раздела

Раздел `SYNC` обеспечивает обмен данными между SkladOptima и внешними каналами: остатки, заказы, метаданные и статусы. Это один из самых критичных разделов продукта, потому что от него зависит точность остатков и отсутствие oversell.

## 2. Целевая аудитория и роли

| Роль | Что хочет получить | Основные действия |
|------|--------------------|-------------------|
| OWNER / ADMIN | Надежную синхронизацию с каналами | Запуск sync, просмотр статусов, диагностика ошибок |
| MANAGER | Понимание состояния обмена данными | Просмотр результата sync |
| Система | Стабильный integration layer | Pull, push, retry, conflict handling |

## 3. Бизнес-цели

- [ ] BG-01: Обеспечить стабильный обмен остатками и заказами.
- [ ] BG-02: Сделать sync наблюдаемым и контролируемым.
- [ ] BG-03: Исключить циклические конфликты и потерю данных.

## 4. Scope раздела

### Что входит

- pull и push сценарии;
- scheduled и ручной sync;
- история sync runs;
- retry и error handling;
- conflict handling;
- статусы и диагностика.

### Что не входит

- произвольный ETL-конструктор;
- пользовательские скрипты синхронизации.

## 5. Пользовательские сценарии

- **US-01:** Как пользователь, я хочу вручную запустить sync.
- **US-02:** Как пользователь, я хочу видеть статус последнего sync.
- **US-03:** Как система, я хочу безопасно повторить failed sync.
- **US-E01:** Что происходит при конфликте данных или старом состоянии канала.

## 6. Функциональные требования

### Must Have

- [ ] FR-01: Система должна поддерживать плановый sync.
- [ ] FR-02: Система должна поддерживать ручной sync из UI.
- [ ] FR-03: Система должна фиксировать историю sync runs.
- [ ] FR-04: Система должна обрабатывать частичные и полные ошибки.
- [ ] FR-05: Система должна защищаться от ping-pong и циклов.

### Should Have

- [ ] FR-10: Система должна поддерживать retry/backoff.
- [ ] FR-11: Система должна показывать item-level ошибки.

### Could Have

- [ ] FR-20: Система должна поддерживать selective sync по типам данных.

## 7. Бизнес-правила и ограничения

- BR-01: Sync не должен разрушать master-данные без явной логики.
- BR-02: Ошибки sync должны быть видимы пользователю.
- BR-03: Конфликты требуют предсказуемой стратегии.

## 8. Пользовательские состояния и ошибки

- Состояние: sync in progress.
- Состояние: success.
- Состояние: partial success.
- Состояние: failed.
- Ошибка: invalid credentials.
- Ошибка: external API timeout.

## 9. Нефункциональные ожидания

| Область | Ожидание |
|---------|----------|
| Надежность | Sync должен быть повторяемым и безопасным |
| Наблюдаемость | Пользователь и support должны видеть статус и ошибки |
| Производительность | Sync не должен блокировать основной UI |
